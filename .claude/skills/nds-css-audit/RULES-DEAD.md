# NDS CSS Audit — Dead Declarations rules (DEAD)

Dead-declaration catalog for `nds-css-audit`. Read during Phase 2/3 of a `DEAD` run (or any run with no rule-group filter). Severity (HIGH/MEDIUM/LOW) drives display order within the DEAD section; byte-delta drives display order across SEL/DEAD/DUPE. Audit flow: `SKILL.md`. Other groups: `RULES-SEL.md` / `RULES-DUPE.md` / `RULES-PERF.md` / `RULES-TOK.md`.

#### CSS Dead declarations (DEAD)

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| DEAD-01 | A **shorthand declaration following a longhand it overwrites** within the same rule body. **Example**: `margin-inline-start: 8px; margin: 0;` — the longhand is dead, the shorthand sets `margin-inline-start: 0` (or implicitly via `margin: 0`).<br><br>Property pairs to check:<br>- `margin-{top,right,bottom,left,inline,block,inline-start,inline-end,block-start,block-end}` → `margin: ...`<br>- `padding-{top,right,bottom,left,inline,block,inline-start,inline-end,block-start,block-end}` → `padding: ...`<br>- `border-{top,right,bottom,left,inline,block,inline-start,inline-end,block-start,block-end}-{width,style,color}` → `border: ...` (and `border-{top,right,bottom,left}: <shorthand>`)<br>- `background-{color,image,position,size,repeat,attachment,origin,clip}` → `background: ...`<br>- `font-{family,size,weight,style,variant,stretch}` and `line-height` → `font: ...`<br>- `flex-{grow,shrink,basis}` → `flex: ...`<br>- `grid-{template-rows,template-columns,template-areas,row-gap,column-gap}` → `grid: ...`<br>- `inset-{block,inline,block-start,...}` → `inset: ...`<br>- `outline-{width,style,color}` → `outline: ...`<br>- `text-decoration-{line,style,color,thickness}` → `text-decoration: ...`<br>- `transition-{property,duration,timing-function,delay}` → `transition: ...`<br>- `animation-{name,duration,...}` → `animation: ...`<br><br>**Explicitly NOT a finding** (fallback / progressive enhancement):<br>- The longhand uses a `var(...)` with **no fallback** while the shorthand uses a literal: `margin-inline-start: var(--my-margin); margin: 0;` — this is an intentional fallback for when `--my-margin` is undefined.<br>- The shorthand precedes the longhand: `margin: 0; margin-inline-start: 8px;` — the longhand wins; this is the correct ordering for an `inline-start` override. NOT a finding.<br>- The two declarations are separated by another declaration that depends on the longhand (rare; investigate). | HIGH | Delete the dead longhand, OR reorder so the longhand follows the shorthand (the intentional-override pattern). **Example**:<br>```scss<br>// Before — dead longhand<br>.x { margin-inline-start: 8px; margin: 0; }<br>// After — option 1: delete dead<br>.x { margin: 0; }<br>// After — option 2: reorder (if the start-override was intended)<br>.x { margin: 0; margin-inline-start: 8px; }<br>```<br>**Compiled bytes saved**: the **minified** longhand declaration (strip leading whitespace + the space after `:`, keep the semicolon). Typically 15–40B compiled. |
| DEAD-02 | The **same property declared two or more times** in the same rule body (excluding custom properties — see DEAD-03) where only the last takes effect.<br><br>**Example**:<br>```scss<br>.x { color: red; color: blue; }  // color: red is dead<br>```<br><br>**Explicitly NOT a finding** (progressive-enhancement fallback):<br>- The first value is a **literal** and the second value uses a `var(--name, <fallback>)` or other newer syntax (e.g., `clamp(...)`, `min(...)`, `max(...)`, `oklch(...)`, `light-dark(...)`). This is the canonical "old browser fallback" pattern.<br>- The first value is a `var()` without fallback and the second value is a literal — also a fallback pattern (when the custom property is undefined, the literal applies).<br>- Annotated with `// DEAD-02 fallback: <reason>` within 2 lines above.<br><br>**Required**: both declarations must be in the **same rule body** at the same nesting level. Declarations in nested blocks (e.g., inside `@include mobile { ... }` or `&:hover { ... }`) are not duplicates of the outer rule.<br><br>**Detection**: parse each rule body, group declarations by normalized property name, flag groups of 2+ that don't match a fallback pattern. | HIGH | Remove the dead (earlier) declaration. **Example**:<br>```scss<br>// Before<br>.x { color: red; color: blue; }<br>// After<br>.x { color: blue; }<br>```<br>**Compiled bytes saved**: the minified dead declaration (typically 10–35B compiled).<br><br>**If the fallback pattern was intended but not annotated**, prefer adding the annotation comment rather than removing the line. |
| DEAD-03 | The **same custom property** (`--name`) defined two or more times in the same rule body. Only the last definition is visible to consumers within the same rule.<br><br>**Example**:<br>```scss<br>.x { --btn-bg: red; --btn-bg: blue; }  // --btn-bg: red is dead<br>```<br><br>**Explicitly NOT a finding**:<br>- Definitions in nested blocks (`@include mobile { ... }`, `&:hover { ... }`) — these are scoped overrides, not duplicates.<br>- Fallback pattern: first definition is a literal, second uses `var(--source, <fallback>)` — annotated with `// DEAD-03 fallback`. Without the annotation, this is hard to distinguish from a true duplicate; require the annotation.<br><br>**Detection**: parse each rule body, find lines matching `--<name>:`, group by name, flag groups of 2+. | HIGH | Keep the last definition; remove the earlier. **Example**:<br>```scss<br>// Before<br>.x { --btn-bg: red; --btn-bg: blue; }<br>// After<br>.x { --btn-bg: blue; }<br>```<br>**Compiled bytes saved**: the minified dead `--name:value;` (typically 15–50B compiled). |
| DEAD-04 | An `@include ltr { ... }` block containing a declaration that is **dead in LTR context** because: (a) the property is a CSS logical property that's already direction-aware (e.g., `text-align: start`, `margin-inline-start`, `padding-inline`, `inset-inline-start`, `border-inline-start-color`, etc.), OR (b) the same property is set to the **same value** in the parent rule (the LTR override is a no-op).<br><br>**Per `CLAUDE.md`**: the `@include ltr` mixin should contain ONLY transforms, gradients, masks, or other properties that logical properties cannot express. Anything inside that has a logical-property equivalent OR re-states the RTL default is dead.<br><br>**Sub-shapes**:<br>- **(a)** `@include ltr { text-align: start; }` — `text-align: start` resolves correctly under both LTR and RTL automatically; the override does nothing.<br>- **(a)** `@include ltr { margin-inline-start: 8px; }` — `margin-inline-start` is already direction-aware; if the parent RTL value is also `8px`, this is identical; if different, the dev should use the parent-rule logical property directly (RTL gets `8px`, LTR gets a different value) — except the logical property only takes ONE value per direction context. If the dev wants different values per direction, they need the physical equivalent (`margin-left`) inside `@include ltr`, NOT the logical equivalent.<br>- **(b)** Parent rule has `color: var(--text-body)` and the `@include ltr` block also has `color: var(--text-body)` — the override is identical to the default; dead.<br><br>**Explicitly NOT a finding**:<br>- `@include ltr { transform: scaleX(-1); }` — transform is direction-asymmetric and has no logical equivalent.<br>- `@include ltr { background-image: linear-gradient(to right, ...); }` — `to right` is direction-aware in LTR but `to inline-end` isn't yet broadly supported on `linear-gradient`; the physical form is correct here.<br>- `@include ltr { margin-left: 0; margin-right: 8px; }` where the parent rule has `margin-inline-start: 8px;` — the parent logical sets `margin-inline-start: 8px` (which in LTR maps to `margin-left: 8px`); the LTR block flips: `margin-left: 0` (resets the start) + `margin-right: 8px` (sets the end). This is a real physical-property override, not dead.<br>- Annotated with `// DEAD-04 keep: <reason>` within 2 lines above.<br><br>**Detection**: find every `@include ltr { ... }` block; for each declaration inside, check (a) whether the property has a logical equivalent that's direction-aware, (b) whether the same property+value pair exists in the parent rule. Flag if either holds. | HIGH | Remove the dead declaration. If the `@include ltr` block becomes empty, remove the block entirely. **Example**:<br>```scss<br>// Before — text-align: start is direction-aware<br>.x {<br>  text-align: start;<br>  @include ltr { text-align: start; }<br>}<br>// After<br>.x {<br>  text-align: start;<br>}<br>```<br>**Compiled bytes saved**: the minified dead declaration (~20B compiled per declaration). **If the block becomes empty post-edit AND the minifier strips empty rules**, also subtract the entire compiled `@include ltr` wrapper: it compiles to `:is(html[dir="ltr"], .ltr) <resolved-selector>{...}`, so the empty wrapper's removal saves `~32B + length-of-resolved-parent-selector` (the `:is(html[dir="ltr"], .ltr) ` prefix is 30 chars + the parent chain + `{}` braces). This is **often larger than the per-declaration savings**; for a typical nested `@include ltr` block, the empty-wrapper savings is 50–120B compiled.<br><br>**Reference**: `CLAUDE.md` "RTL/LTR Support (CRITICAL)" section — `@include ltr` is for transforms, gradients, and other properties logical props don't cover. Use the rule's fix recommendation to nudge the file back toward that contract. |
| DEAD-05 | A **redundant `var(--token-name, <literal-fallback>)` reference** where `--token-name` is unconditionally defined at the top level (i.e., not inside `@media`, `@supports`, or a non-`:root` selector) of one of the three token-source files (`_sass/_variables.scss`, `_sass/_variables-dga.scss`, `_sass/_variables-critical.scss`). The fallback never fires — `var()` always resolves to the token's value — so the `, <literal>` segment is dead bytes AND a maintenance hazard (if the token's value changes upstream, the stale fallback diverges silently and looks intentional in code review).<br><br>**Detection (single-file or full-tree)**:<br>1. Parse the three token-source files. Build the set of "unconditionally defined" token names — definitions at the top of the file or inside `:root { ... }`, NOT inside `@media`, `@supports`, mode selectors (`:root[data-theme="dark"]`, `[data-theme="a11y"]`, etc.), or any other conditional context.<br>2. For each `var(--<name>, <fallback>)` in the target file(s), check whether `--<name>` is in the unconditionally-defined set. If yes, and the fallback is a literal (number, color, keyword) — flag.<br><br>**Why fallback-when-defined is a real hazard, not just bytes**: the redundant fallback masquerades as a defensive default but is actually dead. A future developer sees `var(--spacing-md, 8px)` and assumes `8px` is the canonical value. If `_variables-critical.scss` later changes `--spacing-md` to `12px`, the literal stays `8px` — wrong by inspection, invisible at compile time. The fallback should only exist for tokens that genuinely *might* be undefined (mode-conditional, opt-in, or runtime-injected).<br><br>**Explicitly NOT a finding**:<br>- The token is defined ONLY in a mode override file (`_variables-dark.scss`, `_variables-a11y.scss`, `_variables-dark-dga.scss`, `_variables-dark-ref.scss`) — the base mode has no definition; the fallback IS load-bearing for the base mode.<br>- The token is defined inside a `@media` query, `@supports` block, or a mode selector (e.g., `:root[data-theme="dark"] { --foo: ...; }`) in any source file — conditional definition; the fallback covers the off-condition path.<br>- The fallback is itself a `var()` chain to another token: `var(--token-a, var(--token-b, ...))`. This is a deliberate token-fallback chain pattern (try the more specific token, fall back to the general one). Don't flag.<br>- The fallback is a literal that matches the token's defined value exactly (e.g., `var(--spacing-md, 16px)` where `--spacing-md: 16px;`). Still flag — this is the canonical case the rule targets. The "matches" alignment is what makes it dead.<br>- The token is **not** defined anywhere in the three source files (likely a typo or runtime-injected token) — the fallback IS the only value; don't flag. Separately surface as a gap observation: "undefined token `--<name>` consumed at <file>:<line>".<br>- Annotated with `// DEAD-05 fallback-required: <reason>` within 2 lines above (rare; require a real reason — e.g., "consumed by a host page that may not load NDS variables"). | MEDIUM | Remove the fallback. `var(--token-name, <literal>)` → `var(--token-name)`. **Example**:<br>```scss<br>// `_variables-critical.scss` has `--spacing-md: 16px;` at top level (unconditional).<br>// Before<br>.nds-card { padding: var(--spacing-md, 16px); }<br>// After<br>.nds-card { padding: var(--spacing-md); }<br>```<br>**Compiled bytes saved**: the minified `,<literal>` chunk (the comma+space → comma after minification). Typically 5–15B compiled per occurrence — 1:1 with source character count minus the space.<br><br>**Report format**: for each finding, name the token, cite the file:line where the token is unconditionally defined (so the user can verify the carve-out doesn't apply), and propose the rewrite. |
