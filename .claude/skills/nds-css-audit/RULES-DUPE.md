# NDS CSS Audit — Duplicate Rules / Declarations rules (DUPE)

Duplicate-rule catalog for `nds-css-audit`. Read during Phase 2/3 of a `DUPE` run (or any run with no rule-group filter). Severity (HIGH/MEDIUM/LOW) drives display order within the DUPE section; byte-delta drives display order across SEL/DEAD/DUPE. Audit flow: `SKILL.md`. Other groups: `RULES-SEL.md` / `RULES-DEAD.md` / `RULES-PERF.md` / `RULES-TOK.md`.

#### CSS Duplicate rules / declarations (DUPE)

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| DUPE-01 | **Two non-adjacent rule blocks in the same file** with identical declaration bodies (≥3 declarations) and different selectors.<br><br>Detection: walk the AST, collect every rule's normalized declaration body (sorted properties, normalized whitespace), hash; flag any hash that appears 2+ times. Distance between occurrences doesn't matter — adjacent duplicates may already be candidates for SEL-01 (if selectors differ by a single state), and DUPE-01 catches the cross-section case (a top-of-file rule and a bottom-of-file rule with the same body but different selector roots).<br><br>**Explicitly NOT a finding**:<br>- Adjacent rule blocks already handled by SEL-01 (the SEL-01 finding takes priority; DUPE-01 should not double-flag).<br>- Rule bodies with fewer than 3 declarations — the merge isn't worth the indirection cost.<br>- Bodies that look identical but contain `var(...)` references to component-specific custom properties whose values differ. The bodies are textually identical but resolve to different values — not a true duplicate. Flag only if the custom properties used are shared (defined in `_variables.scss`, not the component).<br>- Bodies inside different media queries (e.g., one inside `@include mobile`, one outside) — the media context makes them non-equivalent.<br><br>**Carve-out**: annotated with `// DUPE-01 split: <reason>` within 2 lines above one of the blocks. | HIGH | Two rewrite options, presented as alternatives:<br><br>**Option A — selector list merge** (when both selectors target similar concerns):<br>```scss<br>// Before<br>.nds-card__title { font-weight: 600; color: var(--text-emphasis); line-height: 1.3; }<br>// ... later ...<br>.nds-alert__heading { font-weight: 600; color: var(--text-emphasis); line-height: 1.3; }<br>// After<br>.nds-card__title, .nds-alert__heading { font-weight: 600; color: var(--text-emphasis); line-height: 1.3; }<br>```<br><br>**Option B — placeholder / mixin extraction** (when both occurrences will likely grow independently):<br>```scss<br>%emphasis-heading { font-weight: 600; color: var(--text-emphasis); line-height: 1.3; }<br>.nds-card__title { @extend %emphasis-heading; }<br>.nds-alert__heading { @extend %emphasis-heading; }<br>```<br><br>Report should present both options; the user picks one. The audit MUST report compiled-byte savings per option separately — Option A and the `@extend %placeholder` form of Option B both save compiled CSS; the `@include mixin` form of Option B does NOT.<br><br>**Compiled bytes saved (Option A — selector-list merge)**: `(minified-body-size × (count − 1))` minus the cost of the joined selector list (the comma-separated selectors are typically smaller than N separate rule blocks with their braces). For a 4-declaration body shared by 2 selectors, typical savings: 60–120B compiled.<br><br>**Compiled bytes saved (Option B — `@extend %placeholder`)**: same as Option A. `%placeholder` emits no CSS on its own; `@extend` collapses all extending sites into a single selector list in compiled output.<br><br>**Compiled bytes saved (Option B' — `@include mixin`)**: **ZERO**. The mixin body inlines at each `@include` call site — compiled CSS is unchanged from the duplicated source. Source readability improves; output size does not. The audit MUST flag this option as a source-organization change, not a byte-savings win. |
| DUPE-02 | **Identical declaration body (≥4 declarations) appearing in ≥3 different component files**. Mixin candidate.<br><br>Detection (full-tree only): hash every rule body across `_sass/components/*.scss`. Flag any hash matching the threshold. Sort findings by `body-size × file-count` (largest output savings first).<br><br>**Mode**: full-tree only. In single-file mode, the audit cannot see other files' rule bodies; DUPE-02 is shown in the "Skipped" footer.<br><br>**Explicitly NOT a finding**:<br>- Bodies fewer than 4 declarations — the mixin overhead exceeds the savings.<br>- Fewer than 3 occurrences across files — promotion to a shared mixin needs at least 3 use sites to justify the indirection cost (matches the JS audit's JSD-05 threshold).<br>- Bodies inside different mixins or media queries — context-dependent duplication is not a candidate for a shared mixin without restructuring the mixin's contract.<br>- The duplicated body already corresponds to an existing mixin in `_sass/_mixins.scss` — the issue is non-use, not absence. Report this as a different sub-finding: "mixin exists but is not used here."<br>- Bodies that use file-local custom properties (`--_<name>`-prefixed or `--<component>-<name>` defined in the component itself, not in `_variables.scss`) — the duplication is incidental; the values resolve differently.<br><br>**Cross-reference**: `_sass/_mixins.scss` currently provides `btn-base`, `btn-focus`, `btn-indicator`, `btn-indicator-styles`, `progress-circle-base/text/progress`, `reset-section-style`, `autofill-reset`. DUPE-02 findings should not duplicate these. | MEDIUM | Propose extracting to `_sass/_mixins.scss` and replacing each occurrence with `@include`. The audit names the mixin (suggest a name based on the first identifying declaration) and proposes the mixin definition and the call sites.<br><br>**Example**:<br>```scss<br>// In _sass/_mixins.scss<br>@mixin focus-ring-emphasis {<br>  outline: 2px solid var(--border-focus);<br>  outline-offset: 2px;<br>  border-color: var(--border-focus);<br>  box-shadow: 0 0 0 4px var(--border-focus-soft);<br>}<br><br>// In _sass/components/_<name>.scss (×3 files)<br>// Before<br>&:focus-visible { outline: 2px solid var(--border-focus); outline-offset: 2px; border-color: var(--border-focus); box-shadow: 0 0 0 4px var(--border-focus-soft); }<br>// After<br>&:focus-visible { @include focus-ring-emphasis; }<br>```<br><br>**Compiled bytes saved** (`@extend %placeholder` form — the byte-saving option): `(minified-body-size × (file-count − 1))` saved. `%placeholder` emits no CSS by itself, and `@extend` collapses all extending sites into a single compiled selector list. For a 5-declaration body shared by 4 files, typical savings: 200–400B compiled.<br><br>**Compiled bytes saved** (`@include mixin` form): **ZERO**. The mixin body inlines at each call site in compiled output — the duplicated bytes remain. Source-organization improvement only. **The audit must NOT propose `@include` as a compiled-byte win.** When proposing extraction, lead with `@extend %placeholder` if compiled size is the goal; offer `@include mixin` as the alternative when the body needs parameters or context-aware logic.<br><br>**Required user judgment**: the audit cannot infer the right mixin/placeholder name from the declarations alone. Present the finding with a suggested name and ask the user to confirm or rename before applying. |
| DUPE-03 | **The same selector appearing 2+ times in the same file at the same nesting level**, with no intervening cascade-relevant rule.<br><br>**Example**:<br>```scss<br>.nds-btn { color: red; }<br>// ... 200 lines of other rules ...<br>.nds-btn { background: blue; }<br>```<br><br>Detection: collect every fully-resolved selector (including parent chain) in the file. For each selector that appears 2+ times, check whether the occurrences are at the same nesting level AND no intervening rule overrides the same properties. If both hold, the duplications could be merged.<br><br>**Explicitly NOT a finding**:<br>- Occurrences at different nesting levels (one inside `@include mobile`, one outside) — context differs.<br>- Occurrences separated by a rule that sets one of the same properties to a different value — the cascade order between them matters.<br>- Annotated with `// DUPE-03 split: <reason>` — readability sectioning is sometimes intentional (e.g., grouping all `:hover` states near each other vs grouping all token definitions near the top).<br>- The selector appears inside an `@media` / `@supports` / `@include mobile()` block — context differs from the outer occurrence.<br><br>**Severity rationale**: LOW because the byte savings are modest (~10–20B per merged occurrence — the duplicate selector text + braces) and the rewrite reorders source in a way that can subtly change developer reading flow. Worth flagging but not worth rushing. | LOW | Merge into one rule, OR annotate the split. **Example**:<br>```scss<br>// Before<br>.nds-btn { color: red; }<br>// ... 200 lines ...<br>.nds-btn { background: blue; }<br>// After<br>.nds-btn { color: red; background: blue; }<br>```<br>**Compiled bytes saved**: the duplicate compiled selector + braces (`<resolved-selector>{}` ~ resolved-selector-length + 2B). Typically 15–40B compiled per merged occurrence depending on selector chain length after `&` resolution.<br><br>**If the split is intentional** for readability/grouping, prefer adding the `// DUPE-03 split: <reason>` annotation comment over merging. |
