# NDS CSS Audit — Token Consistency rules (TOK)

Token-consistency catalog for `nds-css-audit`. Read during Phase 2/3 of a `TOK` run (or any run with no rule-group filter). Severity (HIGH/MEDIUM/LOW) drives display order within the TOK section. TOK findings carry no byte-delta — the impact is design-system integrity (consistent visual language across components, dead-code removal, hierarchy adherence), not output size. Audit flow: `SKILL.md`. Other groups: `RULES-SEL.md` / `RULES-DEAD.md` / `RULES-DUPE.md` / `RULES-PERF.md`.

Per `CLAUDE.md` token hierarchy: **component tokens > semantic tokens > color tokens**. Always prefer the more specific level when one exists. Hex colors and direct `--colors-*` references in components are already enforced elsewhere — not audited here.

**Token-source files (canonical):**
- `_sass/_variables.scss` — NDS component + semantic tokens.
- `_sass/_variables-dga.scss` — DGA (Digital Government Authority) spec reference layer.
- `_sass/_variables-critical.scss` — foundation tokens (spacing, typography, critical-path).

**Mode override files (NOT token sources):** `_variables-dark.scss`, `_variables-a11y.scss`, `_variables-dark-dga.scss`, `_variables-dark-ref.scss`. These re-bind existing tokens for a different mode and should NOT be parsed for new definitions. Consult them only to verify a token has a mode override binding (relevant for TOK-02).

#### CSS Token consistency (TOK)

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| TOK-01 | **Hardcoded numeric value** (e.g., `8px`, `16px`, `1.5rem`, `24px`) appearing in **≥3 component files** where a defined token in any of the three token-source files (`_sass/_variables.scss`, `_sass/_variables-dga.scss`, `_sass/_variables-critical.scss`) has the **same value**.<br><br>Detection (full-tree only):<br>1. Parse the three token-source files: `_sass/_variables.scss`, `_sass/_variables-dga.scss`, `_sass/_variables-critical.scss`. Build a value→token map (resolving `var(--name)` chains to their concrete numeric value). Mode override files (`_variables-dark.scss`, `_variables-a11y.scss`, `_variables-dark-dga.scss`, `_variables-dark-ref.scss`) are NOT parsed for new definitions.<br>2. Walk each component file's declarations. Extract every literal numeric value (`<number><unit>`) used as a property value (not inside `calc()` math, not inside a `var()` fallback).<br>3. Cross-reference: if the literal value has a token equivalent, record the file. After all files are scanned, surface literals that appear in ≥3 different files.<br><br>**Mode**: full-tree only. In single-file mode, the audit cannot see other files' uses; TOK-01 is shown in the "Skipped" footer.<br><br>**Explicitly NOT a finding**:<br>- Values inside `calc()` expressions where the math requires a literal (e.g., `calc(var(--spacing-md) - 2px)`) — the `2px` adjustment is contextual.<br>- Values in `border-width: 1px` and similar low-impact micro-values where the design system may not have a dedicated token.<br>- Values inside an `@media` query condition (`@media (min-width: 768px)`) — those are breakpoint expressions, not consumed properties.<br>- Annotated with `// TOK-01 literal: <reason>` within 2 lines above (rare; require a real reason).<br>- Values inside foundation files (`_variables*.scss`, `_mixins.scss`, `_reset.scss`, `_fonts.scss`) — out of audit scope.<br>- **Decorative offsets, padding, etc., that do not align with the spacing scale** — surface as a separate sub-finding ("not in token table; consider adding") rather than forcing a swap. | HIGH | Replace the literal with the matching token. **Example**:<br>```scss<br>// Token table has --spacing-md: 16px;<br>// Before — in 5 component files<br>.nds-card { padding: 16px; }<br>.nds-alert { padding: 16px; }<br>// ...<br>// After<br>.nds-card { padding: var(--spacing-md); }<br>.nds-alert { padding: var(--spacing-md); }<br>```<br><br>**Report format**: for each duplicated literal, list (a) the matching token name, (b) every file:line where the literal occurs. The user picks per-occurrence or all-at-once. |
| TOK-02 | **Component-level token (`--{component}-…`) that its OWN component does not consume, while a sibling in the same family IS consumed** — a dangling indirection / incomplete feature. NDS is a token *library*, so "defined but unreferenced" is the EXPECTED state for published API and is NOT a finding (see the public-API carve-out).<br><br>Detection (full-tree only):<br>1. Parse the three token-source files: `_sass/_variables.scss`, `_sass/_variables-dga.scss`, `_sass/_variables-critical.scss`. Collect every `--name:` definition — **skip commented-out lines** (`//` or inside a `/* … */` block; a multi-line `/* */` comments out EVERY line until `*/`, not just the line carrying the opener — match the block, not the line prefix).<br>2. For each token name, grep `var(--name` across `_sass/` (excluding the three source files AND the mode override files), `assets/css/*.scss`, `_includes/`, `_layouts/`, **AND the token NAME as a string across `_js/`** — a component can read a token via `getComputedStyle(el).getPropertyValue('--name')`, which a CSS-only grep misses (e.g. `--nds-minimal-nav-bp` is consumed solely by `nds-mainnav.js`; without the `_js/` pass it false-deletes).<br>3. Classify by tier; only a **component-tier** token reaches the sibling test: within its family (`--{component}-{property}-{variant}-*`), is ≥1 sibling consumed by the owning component while THIS member is not? If yes → finding. Cross-check the mode override files (`_variables-dark.scss`, `_variables-a11y.scss`, `_variables-dark-dga.scss`, `_variables-dark-ref.scss`); a mode rebind counts as a use.<br><br>**Mode**: full-tree only. In single-file mode, the audit cannot see other files' uses; TOK-02 is shown in the "Skipped" footer.<br><br>**Explicitly NOT a finding (the published-API carve-out — the decisive one for a token library):**<br>- **Palette tokens** (`--colors-*`), **semantic tokens** (`--background-*`, `--text-*`, `--border-*`, `--icon-*`, `--link-*`, …), and **scale ladders** (`--spacing-*`, `--typo-*`, `--width-*`, `--radius-*`). These ARE the design system's consumable vocabulary — unreferenced internally is their normal, correct state; a deliberately-complete scale/palette has unused rungs by design. Never flag.<br>- **The entire `_variables-dga.scss` mirror** — headered "DGA Design Tokens — DO NOT MODIFY · changes come from upstream." It is a vendored copy of `@platformscode/core`; deleting rungs diverges from upstream and re-appears on the next sync. Report the count for honesty, never recommend a touch.<br>- Tokens referenced **by other tokens** inside the source files (a chain — e.g., `--colors-primary-600` referenced by `--button-background-primary-default`). Don't flag the intermediate; only flag tokens with zero downstream references.<br>- Tokens re-bound by a mode override file — see step 3 above.<br>- Tokens with an explicit `// TOK-02 reserved: <reason>` comment near the definition.<br><br>**Severity rationale**: MEDIUM. The fix is a design decision (complete the feature vs remove the orphan), not a mechanical delete.<br><br>**Motivating example (2026-06-05 full-tree run):** of 248 raw "unreferenced" tokens, 214 were the DGA mirror, 23 were scale ladders, and 11 were NDS palette/semantic API — ALL carved out. Only **4 component orphans** survived the sibling test: `--button-border-{success,warning,info}` (only the `error` sibling is wired into `_buttons.scss`) and `--chip-text-oncolor-disabled` (`_chips.scss` consumes the chip-text family but not this member). | MEDIUM | **Surface as an incomplete-feature review, not a delete.** For each orphan, the owning component consumes siblings but not this member, so the indirection dead-ends and a consumer who sets it sees no effect. The fix is a **design call — complete the feature (add the variant that consumes the token) or remove the orphan.** List each with its definition `file:line` and the consumed siblings. Do NOT auto-delete; never touch palette / semantic / scale / DGA-mirror tokens (carve-out). |
| TOK-03 | **Token usage where a more specific token in the hierarchy exists** for the same effective value.<br><br>**Example**: a component uses `var(--background-surface-elevated)` (semantic-level) where a component-level token `var(--card-background-default)` exists in `_variables.scss` and resolves to the same value via the chain `--card-background-default: var(--background-surface-elevated)`.<br><br>Detection (single-file):<br>1. Read the token table from the three token-source files: `_sass/_variables.scss`, `_sass/_variables-dga.scss`, `_sass/_variables-critical.scss`. Build a dependency graph: each component-level token points to the semantic token it aliases (and through to the color token at the bottom).<br>2. For each `var(--name)` reference in the target file, check whether (a) the referenced token is at the semantic or color level, AND (b) there exists a component-level token in the same component family (e.g., the target file is `_card.scss` and the token table has `--card-*` tokens) that aliases to the same semantic/color token.<br>3. Flag the reference and propose swapping to the component-level token.<br><br>**Explicitly NOT a finding**:<br>- The semantic token is genuinely used cross-component and the value isn't owned by any single component — the semantic level is the right tier.<br>- No component-level token in `_variables.scss` wraps the same value — the semantic is the right tier (you'd need to add the component token first).<br>- The semantic token is being used inside a foundation context (e.g., a card mixin meant to be applied to non-card components) — using the semantic is intentional cross-component sharing.<br>- Annotated with `// TOK-03 cross-component: <reason>` within 2 lines above.<br><br>**Cross-reference**: `CLAUDE.md` "Design Tokens (CRITICAL)" — "always prefer higher levels: 1. Component tokens, 2. Semantic tokens, 3. Color tokens." | MEDIUM | Swap to the more specific component-level token. **Example**:<br>```scss<br>// _variables.scss has:<br>//   --card-background-default: var(--background-surface-elevated);<br>// Before — in _card.scss<br>.nds-card { background: var(--background-surface-elevated); }<br>// After<br>.nds-card { background: var(--card-background-default); }<br>```<br><br>**Why**: the component-level token is the indirection layer that lets the design team retune a single component without affecting every consumer of the semantic token. Bypassing it locks the component to the broader semantic. |
| TOK-04 | **Asymmetric component-token state coverage**. Per `CLAUDE.md`: "New component tokens → add to `_variables.scss`, include all states: default, hovered, pressed, selected, focused, disabled. Add on-color variants for dark backgrounds."<br><br>Detection (full-tree only):<br>1. Parse the three token-source files: `_sass/_variables.scss`, `_sass/_variables-dga.scss`, `_sass/_variables-critical.scss`. Group component-level tokens by family name (e.g., `--button-background-primary-*`, `--card-border-*`).<br>2. For each family, enumerate the state suffixes present: `default`, `hovered`, `pressed`, `selected`, `focused`, `disabled`.<br>3. Flag families that have ≥1 interactive state (`hovered` / `pressed` / `focused`) defined but are missing one or more of the others.<br><br>**Mode**: full-tree only. In single-file mode, the audit cannot see the full token table; TOK-04 is shown in the "Skipped" footer.<br><br>**Explicitly NOT a finding**:<br>- Families that have only `default` (no interactive states defined at all) — the component may genuinely be static (e.g., a `nds-badge` static label).<br>- Families annotated with `// TOK-04 partial-states: <reason>` near the first entry. Some components don't have a pressed state by design (e.g., toggles without a pressed visual), or use shared `:focus-visible` styling from a mixin.<br>- Families covering on-color variants (`--button-background-primary-on-dark-default`, etc.) — these mirror the main family and only need symmetry within their own subset.<br><br>**Severity rationale**: LOW because adding tokens needs design input — the audit can't fill in the colors. This rule surfaces gaps for review rather than for mechanical fix. | LOW | **Surface as a review item**, not an auto-fix. The report lists each family and the missing states. Adding tokens requires:<br>1. Design input on the value (color/spacing/etc.) for each missing state.<br>2. The token added to `_variables.scss` (and the dark-mode equivalent if applicable).<br>3. The component's SCSS updated to consume the new token (replacing inline overrides or removing missing-state behaviors).<br><br>**Example finding**:<br>```<br>### _sass/_variables.scss<br>- L120 [TOK-04] Asymmetric state coverage: --tab-background-primary-*<br>  Defined: default, hovered, focused<br>  Missing: pressed, selected, disabled<br>  Recommendation: review with design; add the missing states to maintain symmetry.<br>```<br><br>Do NOT propose token values; the audit doesn't have the design context. |
| TOK-05 | **Conflicting cross-file duplicate token key, OR an NDS token that hard-copies a value an existing token already owns.** Two classes:<br>(a) **Duplicate key** — the same `--name` defined in >1 source file. DIFFERENT values = the later `@use` silently wins and an import reorder flips it with no error (higher risk); SAME value = redundant restatement.<br>(b) **Value duplication** — an NDS token in `_variables.scss` assigns a raw literal (hex/number) that an existing token already holds (e.g. `--colors-primary-sa-flag-600: #1b8354` duplicating DGA's `--colors-primary-sa-flag-600-primary: #1b8354`); if upstream retints, the copy silently diverges.<br><br>Detection (full-tree only):<br>1. Parse all three source files keyed by `--name`, recording every definition site + value (skip commented-out lines, per TOK-02 step 1).<br>2. (a) Any name with ≥2 live top-level `:root` definitions → flag; tag SAME-value vs DIFFERENT-value.<br>3. (b) Build a value→token map; for each NDS literal-valued token, if another token already holds that exact value, flag as copy-instead-of-alias.<br>4. Resolve the real cascade winner from the `@use` order in the build entry (`assets/css/nds-main.min.scss`) so the report states which value renders today.<br><br>**Mode**: full-tree only.<br><br>**Explicitly NOT a finding**:<br>- **Critical-path literal duplicates** — `_variables-critical.scss` deliberately re-states `--background-default`, `--text-*`, `--shadow-md`, etc. as literals (plus their `:root[data-theme="dark"]` redefs) so critical CSS paints before the alias chain loads. Intentional FOUC prevention.<br>- A duplicate carrying an explicit "DGA uses X, we want Y" override comment AND currently winning the cascade — intentional correction (optionally annotate the load-order dependency).<br>- Mode-scoped redefs (`:root[data-theme="dark"]`, a11y) — different selector, not a true duplicate.<br>- Annotated with `// TOK-05 intentional: <reason>` near the definition. | MEDIUM | **(a)** SAME value → drop the redundant copy, keep the canonical/DGA one. DIFFERENT value → confirm which is intended, then make the override explicit (comment + colocate with the other corrections) so a future `@use` reorder can't silently flip it. **(b)** Replace the hard-copied literal with `var(--canonical-token)` so the value is single-sourced (the editable file loads after DGA, so the alias resolves). Size note: aliasing GROWS compiled bytes (`var(--…)` ≫ a hex) but gzip back-references the recurring prefix → ~0 wire; the win is correctness/anti-drift, not size.<br><br>**Motivating example (2026-06-05, all applied):** aliased `--colors-{primary-sa-flag-600, secondary-gold-600, tertiary-lavendar-500}` to their DGA `*-primary` tokens; documented the `--table-background-hover-selected` override (brand `sa-flag-100` overriding DGA `neutral-50`, mislabeled "missing in DGA"); dropped the duplicate `--icon-neutral` (identical to DGA). |
