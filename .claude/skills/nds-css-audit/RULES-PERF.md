# NDS CSS Audit — Selector Performance / Complexity rules (PERF)

Selector performance and complexity catalog for `nds-css-audit`. Read during Phase 2/3 of a `PERF` run (or any run with no rule-group filter). Severity (HIGH/MEDIUM/LOW) drives display order within the PERF section. PERF findings carry no byte-delta — the impact is selector cost (browser style-matching work) and complexity (maintainability + specificity surprises), not output size. Audit flow: `SKILL.md`. Other groups: `RULES-SEL.md` / `RULES-DEAD.md` / `RULES-DUPE.md` / `RULES-TOK.md`.

#### CSS Selector performance / complexity (PERF)

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| PERF-01 | A **universal selector `*` that is not anchored to a scoped parent**. Modern browser matchers iterate selectors right-to-left, starting from every matched element; `*` at the root of a selector matches every element and forces the matcher to evaluate the rest of the selector for each one. Scoped `*` (e.g., `.nds-card > *`) is bounded; unscoped is not.<br><br>**Detect**:<br>- `* { ... }` at the top level of a partial (matches every element on the page).<br>- `*` as the first compound of a descendant chain with no immediate ancestor anchor (e.g., `* .nds-foo` matches every `.nds-foo` whose ancestor is anything — pointless universal).<br>- `*::before` / `*::after` at the top level (similar to bare `*`).<br><br>**Explicitly NOT a finding**:<br>- **Scoped resets**: `.nds-foo > *` (immediate children of a specific class), `.nds-foo *` (descendants), `.nds-list > * + *` (lobotomized owl spacing inside a scoped list). The universal is bounded by an ancestor.<br>- **Documented border-box reset**: `*, *::before, *::after { box-sizing: border-box; }` in `_sass/_reset.scss` (foundation file — excluded from audit anyway; flagged here for clarity).<br>- Annotated with `// PERF-01 scoped-via-context: <reason>` within 2 lines above. | HIGH | Anchor the universal to a parent class, or remove if redundant. **Example**:<br>```scss<br>// Before — matches every element on the page<br>* { font-family: var(--font-base); }<br>// After — scoped to a component<br>.nds-card * { font-family: var(--font-base); }<br>// Or better: rely on inheritance from a higher-up ancestor<br>.nds-card { font-family: var(--font-base); }<br>```<br><br>**Impact**: in component files, a top-level `*` selector typically means the developer needed inheritance and reached for the universal instead. Inheritance from the component's root is usually correct. |
| PERF-02 | An **unanchored attribute selector** — `[attr]` or `[attr=val]` not preceded by a class or element. Attribute selectors are cheaper than they used to be in modern engines but unanchored attribute matching still walks every element in the relevant subtree. A class anchor (`.nds-foo[disabled]`) restricts the search to elements with the class first.<br><br>**Detect**:<br>- `[hidden] { ... }`, `[disabled] { ... }`, `[aria-hidden="true"] { ... }`, etc. at the top level of a component partial.<br>- `[attr]` as the first compound of a descendant chain with no class/element anchor.<br><br>**Explicitly NOT a finding**:<br>- Inside foundation files (`_reset.scss`, `_base.scss`) where `[hidden]` and similar global-attribute base styles are correct. (These files are excluded from audit; mentioned for clarity.)<br>- Anchored selectors: `.nds-btn[disabled]`, `button[disabled]`, etc. — the anchor restricts the search.<br>- Annotated with `// PERF-02 intentional-global: <reason>` within 2 lines above. The `[hidden]` cross-component anchor is sometimes worth the cost if the goal is a single global rule. | MEDIUM | Anchor the attribute with a class. **Example**:<br>```scss<br>// Before — matches every disabled element on the page<br>[disabled] { opacity: 0.5; cursor: not-allowed; }<br>// After — restricted to NDS components<br>.nds-btn[disabled], .nds-field[disabled] { opacity: 0.5; cursor: not-allowed; }<br>```<br><br>**Alternative**: if multiple components share the disabled-state styling, define a `@mixin disabled-affordance` in `_mixins.scss` and `@include` it in each component. |
| PERF-03 | An **ID selector** (`#foo`) in any `_sass/components/*.scss` file. ID selectors carry specificity weight (`1,0,0`) far above any reasonable component class chain — they make later overrides nearly impossible without further IDs or `!important`. The NDS design system has no need for ID selectors in component styles.<br><br>**Detect**: any selector containing `#<identifier>` in a component file.<br><br>**Explicitly NOT a finding**:<br>- Foundation files (`_grid.scss`, `_base.scss`, `_skeleton.scss`) where global IDs like `#main`, `#header`, `#skip-link` may legitimately appear. (These files are excluded from audit; mentioned for clarity.)<br>- Attribute-selector ID matching: `[id="foo"]` has class-level specificity (`0,1,0`) instead of the ID specificity (`1,0,0`). If a developer must target a specific ID, using the attribute form is acceptable. Flag instead as PERF-04 if the chain is also complex; otherwise no finding.<br>- Annotated with `// PERF-03 specificity-required: <reason>` within 2 lines above. (Extremely rare in a design system; require a real reason.) | HIGH | Convert the ID to a class. **Example**:<br>```scss<br>// Before<br>#nds-special-button { ... }<br>// After<br>.nds-special-button { ... }<br>```<br><br>**If the ID is the only way to target a specific instance in the HTML**, change the markup to add a class instead. ID-targeting is a design-system anti-pattern; the component should not need it. |
| PERF-04 | A **selector with high complexity** by either count of compound selectors (>3 simple selectors joined without combinators, e.g., `.a.b.c.d`) OR descendant-chain depth (>4 levels of nested combinators, e.g., `.a .b > .c .d ~ .e`). Selector resolution must include `@include` expansion from `_sass/_mixins.scss` — if a mixin like `@include ltr` wraps `@content` in `:is(html[dir="ltr"], .ltr) <selector>`, the resolved selector at that emit point includes the wrapper.<br><br>**Detect**:<br>- **Compound count**: split each selector on combinators (` `, `>`, `+`, `~`); for each resulting compound, count `.`/`#`/`[attr]` + `::pseudo-element` + `:pseudo-class` segments. If any compound has >3, flag.<br>- **Descendant depth**: count combinators in the resolved selector (including `&` resolution under nested rules AND mixin-emitted selector wrappers). If >4, flag.<br><br>**Explicitly NOT a finding**:<br>- Pseudo-class chains where most chains are state pseudo-classes (`:hover`, `:focus-visible`, `:disabled`, `:checked`, `:read-only`) — counting these as "compound complexity" over-flags legitimate state-aware selectors. Cap pseudo-class contribution at 2 per compound.<br>- Annotated with `// PERF-04 complexity-required: <reason>` within 2 lines above.<br><br>**Severity rationale**: MEDIUM because the cost is gradual — a 4-deep chain isn't catastrophic, but it accumulates across the bundle and signals a maintainability issue (specificity hard to override, hard to read). | MEDIUM | **Type:** `tradeoff → recommend solving` (no purely mechanical fix; the audit MUST still propose a concrete path with cost framing rather than punting to "review with design").<br><br>For each PERF-04 finding, the report MUST include:<br>- **Proposed solution:** one specific path the audit recommends, with the SCSS rewrite sketched.<br>- **Benefit vs cost:** what this saves (compiled bytes, specificity reduction, override-ability) vs what it requires (markup changes, redesign of the variant axis, etc.).<br>- **Alternative:** one runner-up option, briefly.<br>- **When to accept and annotate:** if the cost outweighs the benefit (deliberate variant grid encoding 3+ compositional axes, the `:not()` guard is doing real work and can't be hoisted), recommend the annotation-exemption (`// PERF-04 complexity-required: <reason>`) and move on. Acceptance IS a valid recommendation when restructuring is more expensive than the cost.<br><br>**Solution preference per the project's size > perf priority** (see SKILL.md "Size vs Perf"):
- **Solution B (flatten nesting)** — recommended whenever the finding is depth-driven. Pure win on perf + readability, zero size cost.
- **Accept-and-annotate** — recommended whenever the finding is compound-count-driven on a variant grid or other legitimate composition. Zero size cost, zero perf cost.
- **Solution A (collapse axis into single class)** — **DO NOT RECOMMEND.** This grows the bundle (size ↑) for a sub-millisecond perf gain. Per size > perf, the audit explicitly de-recommends Solution A. Document it as the alternative for posterity, but never as the recommendation.

**Two solution shapes** (pick the one that fits the finding; never both):<br><br>**Solution A — collapse a compositional axis into a single class.** Use when the compound expresses N orthogonal axes (variant × style × guard) and the compound count > 3 comes from composing them. Mint a single class per combination. Drops compound count by `N-1`, requires markup changes wherever the compositional classes appear.<br>```scss<br>// Before (4-compound: variant × style × guard × base)<br>.nds-card:not(.nds-oncolor, [disabled]).nds-gold.nds-color { /* ... */ }<br>// After (single class per combo): mint .nds-card-fill-gold; drop the variant nesting<br>.nds-card-fill-gold:not(.nds-oncolor, [disabled]) { /* ... */ }<br>// HTML: `class="nds-card nds-gold nds-color"` → `class="nds-card nds-card-fill-gold"`<br>```<br>**Size impact:** **+N B compiled** — each of the N new classes adds its own rule block to the bundle. For 12 combos (6 colors × 2 modes), the added selector definitions are typically larger than the saved per-emit-point segment bytes. **Solution A grows the bundle.**<br>**Perf impact:** **improves** — compound count drops from 4 to 3 at every variant emit point (browsers do less work per element). Bounded but real, especially for elements that match many variants on the same page.<br>**Tradeoff:** classic perf-for-size swap. Best when (a) the variant grid is large enough (≥6 variants) that the per-emit perf cost compounds across pages, AND (b) the markup change is one-time and tooling-assisted. Worst when the grid is small or markup is hand-authored across many repos.<br><br>**Solution B — flatten the SCSS nesting.** Use when the >3 compound is just an artifact of nested `&` chains. Drops descendant-chain depth without changing compound count or markup.<br>```scss<br>// Before (5-deep descendant chain from nesting)<br>.nds-card { .nds-card__header { .nds-icon { &.nds-icon-emphasis { &.nds-icon-large { /* ... */ } } } } }<br>// After (2-deep, single 3-compound)<br>.nds-card__header .nds-icon.nds-icon-emphasis.nds-icon-large { /* ... */ }<br>```<br>**Size impact:** **±0 B compiled** — the selector emits identically after `&` resolution. Source is shorter to read, but the compiler produces the same output.<br>**Perf impact:** **improves** — descendant-chain depth drops; specificity often lowers too, making overrides easier.<br>**Tradeoff:** none — pure win on perf and source readability, zero cost on size. Always preferable to Solution A when the >3 issue is purely depth, not orthogonal axes.<br><br>**Worked example — variant-grid `tradeoff → accept-and-annotate` case** (the L383 `.nds-card:not(.nds-oncolor, [disabled]).nds-gold.nds-color` pattern):<br><br>The finding compound is `.nds-card:not(.nds-oncolor, [disabled]).nds-gold.nds-color` — 4 segments encoding **base × oncolor-guard × variant-color × variant-style**. Solution A would mint 12+ classes (`.nds-card-fill-{gold,neutral,lavender,yellow,red,blue}-{base,dark}`) and require every card consumer to swap markup. Solution B doesn't apply — the depth isn't the issue.<br><br>**Proposed solution:** *accept and annotate IF AND ONLY IF the size+perf scoring confirms acceptance is the right call*. The PERF-04 finding has these typical impact profiles for this pattern:<br>- **Solution A** (mint per-combo classes): size **+250 to +600B compiled** (12+ new selector definitions), perf **+small** (compound 4→3 at 12 emit points). Tradeoff.<br>- **Solution B** (flatten nesting): N/A — the >3 issue is compound count, not depth.<br>- **Accept-and-annotate**: size **±0**, perf **±0**. Closes the finding via comment; the variant grid's compositional cost stays on the wire.<br><br>For the L383 cards.scss case specifically: Solution A's size cost (~+400B for 12 new classes) exceeds the bounded perf impact (4-vs-3 segment match at 12 emit points is sub-millisecond on real workloads). **Accept-and-annotate is the right call here** because the tradeoff resolves toward keeping bytes off the bundle.<br><br>Add this comment above the variant block:<br>```scss<br>// PERF-04 variant-grid: deliberate 4-axis composition (base × oncolor-guard × variant-color × variant-style); restructuring to Solution A would add ~12 class definitions to the bundle for a sub-ms per-match improvement. Tradeoff resolves toward keeping bytes off the wire.<br>```<br>Future runs see the annotation and skip the rule.<br><br>**However:** PERF-04's compound-count finding being annotated does NOT mean Cards is "done." The variant grid has a SECOND structural opportunity the current rules don't catch in their current form: the 12 variant emit points (6 colors × {base, dark}) likely SHARE 70-90% of their declaration bodies. If they do, that's a multi-site DUPE-01 / DUPE-02 candidate that crosses the variant-grid boundary — a real size win without any perf cost. The next run should re-scan the variant block with DUPE eyes after the annotation lands, and the gap inventory should explicitly track this.<br><br>**Alternative:** Solution A becomes the right call IF the variant grid grows beyond 8-9 colors (the per-emit-point perf cost starts compounding faster than the bundle bloat). Revisit then.<br><br>**Impact (general)**: each level of nesting/compound adds work to the browser's selector-matching pass AND raises specificity, making future overrides require equally-specific selectors. |
