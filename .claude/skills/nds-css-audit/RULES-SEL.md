# NDS CSS Audit — Selector Compaction rules (SEL)

Selector-compaction catalog for `nds-css-audit`. Read during Phase 2/3 of a `SEL` run (or any run with no rule-group filter). Severity (HIGH/MEDIUM/LOW) drives display order within the SEL section; byte-delta drives display order across SEL/DEAD/DUPE. Audit flow: `SKILL.md`. Other groups: `RULES-DEAD.md` / `RULES-DUPE.md` / `RULES-PERF.md` / `RULES-TOK.md`.

#### CSS Selector compaction (SEL)

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| SEL-01 | Two or more rule blocks within the **same parent scope** (same nesting level under the same parent selector) whose selectors differ only by a single pseudo-class, attribute, or `&`-prefixed token, AND whose declaration bodies are **identical** (same property-value pairs, same order, ignoring whitespace).<br><br>Detection: walk the AST of each component file; for each parent block, group children by normalized declaration body; flag groups of 2+ children whose selectors all share the same base. Common shapes: `&:hover` + `&:focus-visible`, `&[data-state~="hover"]` + `&:hover`, `&[aria-selected="true"]` + `&.is-selected`.<br><br>**Explicitly NOT a finding**:<br>- Blocks already grouped via `&:is(...)` (the rewrite has already been applied).<br>- Declaration bodies that differ in any property or value, even by one custom property — the merge would change cascade.<br>- Selectors that include a combinator change (e.g., one is `&:hover` and another is `& > .child:hover`) — the merge would change scope.<br>- Blocks separated by an intervening rule with a different selector and overlapping property — the source order is load-bearing for the cascade.<br><br>**Carve-out (annotation exemption)**: a `// SEL-01 split: <reason>` comment within 2 lines above one of the blocks indicates the split is intentional (e.g., readability sectioning for a long file, or a documented future divergence). | HIGH | Combine the matching selectors into a single `&:is(<a>, <b>, <c>)` block. The CSS specification's `:is()` reports the highest specificity of its arguments, which is what we want here. Browser support: Safari 14+, Chrome 88+, Firefox 78+ — all evergreen. **Example**:<br>```scss<br>// Before<br>&:hover { color: var(--x); border-color: var(--y); }<br>&:focus-visible { color: var(--x); border-color: var(--y); }<br>// After<br>&:is(:hover, :focus-visible) { color: var(--x); border-color: var(--y); }<br>```<br>**Compiled bytes saved**: `(resolved-selector-chain + minified-body) × (count − 1) − :is(<states>)-overhead`. The **resolved selector chain** is the full compiled selector after `&` resolution (e.g., `&:hover` nested under `.nds-card .nds-header` resolves to `.nds-card .nds-header:hover` ~ 30B). Source-character estimates severely understate this when the rule is nested. With N=2 merges and a 3-decl body, typical savings: 60–150B compiled.<br><br>**Reference good pattern**: `_sass/components/_buttons.scss` already uses `&:is(:hover, [data-state~="hover"])` in several places — match that style. |
| SEL-02 | A selector containing **two or more chained `:not(...)` calls** on the same target. Detection regex: `:not\([^)]+\):not\(`. Matches `&:not(:disabled):not(:read-only)`, `.foo:not([hidden]):not([aria-disabled="true"])`, etc.<br><br>**Explicitly NOT a finding**:<br>- A single `:not(...)` call (already optimal).<br>- A `:not(...)` containing a complex selector list that the modern `:not(a, b)` form would reject — if the inner selectors are themselves compound (e.g., `:not(.foo .bar)`), check browser support before rewriting; combining is still safe in evergreen browsers.<br><br>**Browser support note**: `:not(a, b, c)` (selector list argument) — Safari 9+, Chrome 88+, Firefox 84+. All evergreen. | MEDIUM | Combine into one `:not(a, b, c)` call. **Example**:<br>```scss<br>// Before<br>&:not(:disabled):not(:read-only):not([aria-disabled="true"]) { ... }<br>// After<br>&:not(:disabled, :read-only, [aria-disabled="true"]) { ... }<br>```<br>**Compiled bytes saved**: ~6B per `:not(` collapsed (1:1 with source — minifier preserves these literally). |
| SEL-03 | An **over-qualified type selector** — a compound selector of the form `<tag>.nds-<name>` (e.g., `button.nds-btn`, `a.nds-link`, `input.nds-field`) where `.nds-<name>` appears **nowhere else in the file** paired with a different element. The type qualifier adds no discrimination because the class is only ever used on one element type.<br><br>Detection: collect every `<tag>.nds-<name>` selector in the file. For each, check whether `.nds-<name>` appears (a) on a different tag, or (b) as a bare class anywhere else in the file. If both checks pass (the class is only ever on this one tag, AND the bare class doesn't appear), the type qualifier is dead.<br><br>**Explicitly NOT a finding**:<br>- Intentional element discrimination: `input.nds-field` AND `textarea.nds-field` both appear — the type qualifier is doing work.<br>- The bare `.nds-<name>` appears as a state hook on a different element (e.g., `.nds-modal.is-open .nds-field` — the field can appear in modal contexts as just `.nds-field`).<br>- Specificity is being deliberately raised — comment annotation `// SEL-03 specificity: <reason>` exempts.<br><br>**Cross-file consideration**: this rule only checks within a single file. If the class is used on other elements in other component files, the type qualifier is still load-bearing and the local file's rewrite would break cascade elsewhere. Note this caveat in the report. | MEDIUM | Drop the type qualifier. **Example**:<br>```scss<br>// Before<br>button.nds-btn { padding-inline: 12px; }<br>// After<br>.nds-btn { padding-inline: 12px; }<br>```<br>**Compiled bytes saved**: tag-length × number of compiled selector occurrences (each emit point after `&` resolution). `button.nds-btn` resolved to 5 sites compiles to 5 × 6B = 30B saved.<br><br>**Note** to include in the rewrite: "Verify across other component files that `.nds-<name>` is not used on a different tag — this rewrite drops specificity, and a cascade conflict could surface." |
| SEL-04 | A **comma-separated selector list** of 3+ entries where every entry shares a common prefix and `:is()` would compact. **Example**: `.parent .child-a, .parent .child-b, .parent .child-c { ... }`. Detection: within a parent block, find rule selectors that are comma-separated lists, parse the entries, find the longest common prefix; if the prefix is non-trivial and the suffixes are short (each ≤ 3 simple selectors), flag for `:is()` rewrite.<br><br>**Explicitly NOT a finding**:<br>- Lists where the common prefix is empty or just `&`.<br>- Lists where any entry would lose specificity via the rewrite — `:is()` reports the highest specificity of its arguments, so combining a `.foo` and `.foo .bar` would change the latter's specificity.<br>- Lists where the entries are already short and adding `:is(...)` would lengthen the selector. The whole point is byte savings; if the rewrite is longer, skip.<br><br>**Severity rationale**: byte savings are typically small (5–20B per occurrence) and the rewrite changes specificity behavior subtly — LOW so it doesn't dominate the report. | LOW | Rewrite as `<prefix>:is(<suffix-a>, <suffix-b>, <suffix-c>)`. **Example**:<br>```scss<br>// Before<br>.nds-card .nds-title, .nds-card .nds-subtitle, .nds-card .nds-meta { color: var(--text-muted); }<br>// After<br>.nds-card :is(.nds-title, .nds-subtitle, .nds-meta) { color: var(--text-muted); }<br>```<br>**Compiled bytes saved**: typically 5–20B per occurrence after `&` resolution and minification. **Often near-zero or negative once `:is(...)` overhead is counted** — skip the finding if the rewrite is not strictly shorter post-resolution. **Always verify** the entries' original specificities are equivalent. |
