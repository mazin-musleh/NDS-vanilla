# NDS CSS Audit — Selector Compaction rules (SEL)

Selector-compaction catalog for `nds-css-audit`. Read during Phase 2/3 of a `SEL` run (or any run with no rule-group filter). Severity (HIGH/MEDIUM/LOW) drives display order within the SEL section; byte-delta drives display order across SEL/DEAD/DUPE. Audit flow: `SKILL.md`. Other groups: `RULES-DEAD.md` / `RULES-DUPE.md` / `RULES-PERF.md` / `RULES-TOK.md`.

**Gzip-awareness — SEL barely shrinks the wire.** SEL collapses *repeated* content: duplicated declaration bodies (SEL-01), recurring tokens like `:not(` (SEL-02), repeated selector chains (SEL-03). That is exactly what gzip/Brotli already compress — a repeat within the window becomes a ~3-byte back-reference regardless of its length. So the **raw compiled-byte** savings these rules can compute *severely overstate* the **wire** savings: the gzip-discounted delta is roughly the entropy of the *differing* token, not the byte count of the collapsed text. (Same lesson as the deleted DUPE-04 rule — "Brotli reduced compiled savings to ~50–100B.")

Therefore, for every SEL finding: **lead with the structural benefit** — fewer compiled rules / parse cost (SEL-01), selector match-cost (SEL-02), specificity reduction (SEL-03) — and report wire-bytes as a *gzip-discounted secondary* figure, never the headline. **SEL is a cleanliness / specificity / parse-cost group, not a byte-reduction group.** When the goal is shrinking the gzipped wire, the levers are DEAD (remove declarations) and TOK-02 (remove dead tokens) — they delete *unique* content gzip cannot reconstruct.

#### CSS Selector compaction (SEL)

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| SEL-01 | Two or more rule blocks within the **same parent scope** (same nesting level under the same parent selector) whose selectors differ only by a single pseudo-class, attribute, or `&`-prefixed token, AND whose declaration bodies are **identical** (same property-value pairs, same order, ignoring whitespace).<br><br>Detection: walk the AST of each component file; for each parent block, group children by normalized declaration body; flag groups of 2+ children whose selectors all share the same base. Common shapes: `&:hover` + `&:focus-visible`, `&[data-state~="hover"]` + `&:hover`, `&[aria-selected="true"]` + `&.is-selected`.<br><br>**Explicitly NOT a finding**:<br>- Blocks already grouped via `&:is(...)` (the rewrite has already been applied).<br>- Declaration bodies that differ in any property or value, even by one custom property — the merge would change cascade.<br>- Selectors that include a combinator change (e.g., one is `&:hover` and another is `& > .child:hover`) — the merge would change scope.<br>- Blocks separated by an intervening rule with a different selector and overlapping property — the source order is load-bearing for the cascade.<br>- Blocks inside a mode-override scope (`@at-root` oncolor/dark/a11y, `:root[data-theme~="dark"] …`) where the corresponding light/default sibling block (same selector pair, same parent) uses **distinct** values for the same two states. The override is collapsing a missing rung onto an existing one (e.g. `selected` falling back onto `pressed` because no `-selected-oncolor` token exists), not a true merge candidate — surface as TOK-04 (missing family rung) instead.<br><br>**Carve-out (annotation exemption)**: a `// SEL-01 split: <reason>` comment within 2 lines above one of the blocks indicates the split is intentional (e.g., readability sectioning for a long file, or a documented future divergence). | HIGH | Combine the matching selectors into a single `&:is(<a>, <b>, <c>)` block. The CSS specification's `:is()` reports the highest specificity of its arguments, which is what we want here. Browser support: Safari 14+, Chrome 88+, Firefox 78+ — all evergreen. **Example**:<br>```scss<br>// Before<br>&:hover { color: var(--x); border-color: var(--y); }<br>&:focus-visible { color: var(--x); border-color: var(--y); }<br>// After<br>&:is(:hover, :focus-visible) { color: var(--x); border-color: var(--y); }<br>```<br>**Primary win: one fewer compiled rule** (less to parse and style-match) + source clarity — lead the finding with this, not bytes (see the gzip preamble). **Wire bytes (gzip-discounted): SMALL** — the duplicated body and repeated selector chain are already back-referenced by gzip, so the raw `(chain + body) × (N−1)` figure overstates the wire delta; the real saving is ≈ the removed back-reference overhead minus the `:is(<states>)` cost.<br><br>**Reference good pattern**: `_sass/components/_buttons.scss` already uses `&:is(:hover, [data-state~="hover"])` in several places — match that style. |
| SEL-02 | A selector containing **two or more chained `:not(...)` calls** on the same target. Detection regex: `:not\([^)]+\):not\(`. Matches `&:not(:disabled):not(:read-only)`, `.foo:not([hidden]):not([aria-disabled="true"])`, etc.<br><br>**Explicitly NOT a finding**:<br>- A single `:not(...)` call (already optimal).<br>- A `:not(...)` containing a complex selector list that the modern `:not(a, b)` form would reject — if the inner selectors are themselves compound (e.g., `:not(.foo .bar)`), check browser support before rewriting; combining is still safe in evergreen browsers.<br><br>**Browser support note**: `:not(a, b, c)` (selector list argument) — Safari 9+, Chrome 88+, Firefox 84+. All evergreen. | MEDIUM | Combine into one `:not(a, b, c)` call. **Example**:<br>```scss<br>// Before<br>&:not(:disabled):not(:read-only):not([aria-disabled="true"]) { ... }<br>// After<br>&:not(:disabled, :read-only, [aria-disabled="true"]) { ... }<br>```<br>**Primary win: selector match-cost** — one `:not()` to evaluate per match instead of a chain — plus clarity. **Wire bytes (gzip-discounted): ~0** — `:not(` recurs throughout the sheet, so gzip already back-references it; the raw ~6B-per-collapse is essentially absent on the wire. Apply for match-cost/clarity, not bytes. |
| SEL-03 | An **over-qualified type selector** — a compound selector of the form `<tag>.nds-<name>` (e.g., `button.nds-btn`, `a.nds-link`, `input.nds-field`) where `.nds-<name>` appears **nowhere else in the file** paired with a different element. The type qualifier adds no discrimination because the class is only ever used on one element type.<br><br>Detection: collect every `<tag>.nds-<name>` selector in the file. For each, check whether `.nds-<name>` appears (a) on a different tag, or (b) as a bare class anywhere else in the file. If both checks pass (the class is only ever on this one tag, AND the bare class doesn't appear), the type qualifier is dead.<br><br>**Explicitly NOT a finding**:<br>- Intentional element discrimination: `input.nds-field` AND `textarea.nds-field` both appear — the type qualifier is doing work.<br>- The bare `.nds-<name>` appears as a state hook on a different element (e.g., `.nds-modal.is-open .nds-field` — the field can appear in modal contexts as just `.nds-field`).<br>- Specificity is being deliberately raised — comment annotation `// SEL-03 specificity: <reason>` exempts.<br><br>**Cross-file consideration**: this rule only checks within a single file. If the class is used on other elements in other component files, the type qualifier is still load-bearing and the local file's rewrite would break cascade elsewhere. Note this caveat in the report. | MEDIUM | Drop the type qualifier. **Example**:<br>```scss<br>// Before<br>button.nds-btn { padding-inline: 12px; }<br>// After<br>.nds-btn { padding-inline: 12px; }<br>```<br>**Primary win: specificity reduction** — `<tag>.nds-name` is `0,1,1`; `.nds-name` is `0,1,0`, easier to override and less `!important` pressure downstream. **Wire bytes (gzip-discounted): ~tag-length once** — gzip back-references the repeated `<tag>.nds-name` after the first occurrence, so the raw `6B × 5 = 30B` is really ~6B on the wire. The win is the specificity drop, not the bytes.<br><br>**Note** to include in the rewrite: "Verify across other component files that `.nds-<name>` is not used on a different tag — this rewrite drops specificity, and a cascade conflict could surface." |

#### Deleted rules & declined patterns (lessons — do not re-introduce)

Preserved decisions, not active rules. The audit must NOT re-propose them.

- **SEL-04 (DELETED) — comma-list → `:is()` compaction.** Flagged a comma-separated selector list sharing a common prefix (`.p .a, .p .b, .p .c` → `.p :is(.a, .b, .c)`). Deleted: the raw saving was already marginal (5–20B), the shared prefix is gzip-back-referenced so the WIRE saving is ~0, the `:is()` overhead often made the rewrite net-**negative** on bytes, and the rule's own guidance was "skip if not strictly shorter" — i.e. it mostly produced non-findings. Same profile as the deleted DUPE-04 rule. Its specificity-equivalence check is subsumed by SEL-01's. **Do not re-introduce a comma-list `:is()` byte-compaction rule.**
