# NDS JS Audit — Security rules (JSS)

Client-side web-security sink catalog for `nds-js-audit`. Read during Phase 2/3 of a `security` run (or any run with no rule-group filter). Severity is calibrated to exploitability in a Jekyll static-site context. Audit flow: `SKILL.md`. Other groups: `RULES-JSP.md` / `RULES-JSD.md` / `RULES-JSA.md`.

#### JS Security (JSS)

Web-security sinks that commonly appear in vanilla-JS component code. Severity is calibrated to exploitability in a Jekyll static-site context: a reflected XSS through an NDS component is HIGH because the component ships everywhere; a same-origin `window.open` with a developer-controlled `data-*` attribute is MEDIUM. Each JSS finding must name the untrusted-input source in plain language — "this `${title}` comes from `NDSAlert.create()` options, which callers can fill with arbitrary strings" — so the fix cost is visible alongside the risk.

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| JSS-01 | Direct HTML-sink assignment or method call where the RHS / argument is a template literal or `+` concatenation containing at least one non-literal substitution: `el.innerHTML = X`, `el.outerHTML = X`, `el.insertAdjacentHTML(pos, X)`, `document.write(X)`, `document.writeln(X)`, `<documentRef>.write(X)`, `<documentRef>.writeln(X)` — where `<documentRef>` resolves to a `Document` (most commonly `iframe.contentDocument`, `popup.document`, or `globalThis.document`). Classic XSS. Focus on sinks that interpolate caller-supplied arguments (component-options, `dataset` reads, API responses, `getAttribute`), not sinks where every substitution is compile-time-known. The `<documentRef>.write` shape covers the iframe-init idiom (build full `<!doctype html>` document, hand to `iframe.contentDocument.write`) where DOM construction via `createElement` is impractical — the same escape rules apply regardless of which sink delivers the markup. Motivating example for the doc-write shape: `_js/nds-export.js:284-294` builds a print iframe via `doc.write(html)` with every substitution routed through `NDS.escapeHtml` (currently safe; the rule's prior sink list missed this shape).<br><br>**Explicitly NOT a finding**:<br>- Template literals where every `${}` substitution matches the numeric/boolean allowlist. A substitution is allowlisted when, and ONLY when, it is one of:<br>  **(a)** identifier from a visible `for`-loop counter (`i`, `j`, `k`, `n`, `index`, `idx`, or the loop's declared counter name);<br>  **(b)** call expression `parseInt(...)` / `parseFloat(...)` / `Number(...)` / `Math.<any>(...)`, OR `.toString()` / `.padStart(n, '0')` / `.padEnd(n, '0')` / `.toFixed(n)` / `.toLocaleString(...)` on a numeric-by-allowlist receiver — these can only return digit-strings;<br>  **(c)** numeric literal (`42`, `3.14`, `0xFF`);<br>  **(d)** `.length` property of any expression;<br>  **(e)** boolean literal (`true`, `false`);<br>  **(f)** arithmetic / bitwise / comparison / logical expression (`+`, `-`, `*`, `/`, `%`, `**`, `<`, `>`, `&`, `\|`, `&&`, `\|\|`) whose operands all independently match (a)–(e) or (f) recursively;<br>  **(g)** a variable whose most recent assignment in the same function scope resolves to (a)–(f), OR is a template literal whose every `${}` substitution independently matches (a)–(j). One non-matching substitution forfeits the whole variable;<br>  **(h)** ternary `cond ? A : B` where BOTH `A` and `B` are string literals containing no `${}` substitutions — the value is one of two compile-time-known strings;<br>  **(i)** reference to a module-scope `const` declared in the same file with a compile-time-known literal initializer. Three sub-shapes:<br>    - **(i.1)** direct reference `${CONST}` where initializer is a single string/number literal;<br>    - **(i.2)** property access `${CONST.key}` / `${CONST[expr]}` where initializer is an object literal whose every leaf is a string/number literal (nested objects allowed);<br>    - **(i.3)** element access `${CONST[i]}` where initializer is an array literal whose every element is a string/number literal.<br>    Any non-literal leaf in the initializer (function call, dynamic import, template literal with substitutions, variable reference) disqualifies the exemption;<br>  **(j)** instance/object property read `<obj>.<prop>` (or chained `<obj>.<prop1>.<prop2>`) where EVERY assignment to `<prop>` anywhere in the same file resolves to (a)–(f) recursively. Whole-file scope. Covers by-construction numeric fields (`this.totalSteps`, `this.currentStep`, `this.activeIndex`). A single assignment outside (a)–(f) anywhere in the file disqualifies the property entirely — partial coverage doesn't count. Disqualifying chains: `this.opts.colors[i]`, `getAttribute` / `dataset` / `localStorage` reads, non-allowlisted function-call returns.<br>- Interpolations wrapped in the file's own escape helper (`${escapeHtml(x)}` / `${encodeEntities(x)}` / equivalent defined in the same file).<br>- Sinks where the RHS is a static string with no `${}` substitutions.<br>- Annotation exemption (global) naming the specific source of the value AND why it is safe (e.g., `// XSS-safe: icon class comes from TYPE_DEFAULTS keys, not user input`).<br>- Sinks on freshly-created elements where the value was already `.textContent`-assigned elsewhere (handled as a combined read).<br><br>**Critical override — taint-source escalation.** None of the exemptions above apply when the value's nearest assignment chain reads from a same-origin-writable store: `localStorage.getItem(...)` / `sessionStorage.getItem(...)` / `document.cookie` / a same-file helper wrapping any of these. Same-origin XSS in another page, malicious extensions with `storage` access, or dev-tools paste can plant attacker-controlled HTML; the next render injects it. The taint-source escalation is a HIGH finding regardless of which exemption otherwise applied. Detection: trace the RHS variable's most recent assignment; if it terminates at a writable-store read (direct or via same-file helper), fire. Fix is `NDS.escapeHtml(cached)`, OR migrate the producer to write `.textContent`-safe data only, OR an annotation naming the store AND why the cached payload is trusted (e.g., "// XSS-safe: cache writer at L88 escapes via NDS.escapeHtml before storing"). Motivating example (resolved): `_js/nds-cityWeather.js` formerly interpolated localStorage-cached payloads into HTML sinks; it now renders them via `.textContent` with the rationale documented in the comment at the top of its IIFE. The escalation stands as a regression guard. | HIGH | **Primary**: wrap every caller-supplied substitution in `${NDS.escapeHtml(x)}` — escapes via the browser's textContent serializer, catches edge cases (quotes, backticks, entities, attribute-context bypasses) a regex escape misses. One-line per substitution. **Attribute-context values** still need `setAttribute` rather than interpolation — `escapeHtml` doesn't close quote-breakout vectors in attribute values; lift those out of the template. **Secondary** (mixed-chrome templates / avoid innerHTML entirely): build DOM via `document.createElement` and assign dynamic values through `.textContent` / `.setAttribute`. **Never use regex-based escapes.** |
| JSS-02 | Dynamic code execution sinks: `eval(x)`, `new Function(...args, body)`, `setTimeout(x, ...)` / `setInterval(x, ...)` where the first argument is a string literal or a string-valued variable (not a function reference, arrow, or function expression). The browser will parse and execute the argument as JavaScript, bypassing every escape upstream and skating past any CSP `script-src` restriction that doesn't include `'unsafe-eval'`. There is no legitimate use case for any of these in NDS component code. | HIGH | Replace with a direct function call. `setTimeout('foo()', 100)` → `setTimeout(foo, 100)`. For dynamic dispatch (action name → handler), use a lookup object: `({ next: goNext, prev: goPrev }[action] ?? noop)()`. If you genuinely need to parse user-supplied expressions (extremely rare for UI code), sandbox it in a Worker with a dedicated parser — not in the main realm via `eval`. |
| JSS-03 | Anchor element with `target="_blank"` (or a template-literal expression that could resolve to `_blank` at runtime) and NO `rel="noopener"` or `rel="noreferrer"`, in three sub-shapes: **(a)** HTML string literals containing `<a ... target="_blank" ...>` where no `rel=` attribute in the same tag includes `noopener` or `noreferrer`; **(b)** DOM construction — `document.createElement('a')` followed by `.target = '_blank'` or `.setAttribute('target', '_blank')` with no matching `.rel =` / `setAttribute('rel', ...)` in the same setup block; **(c)** HTML string templates where the `target` attribute value is set via any `${...}` substitution — including conditional spread patterns like `${cond ? \` target="${value}"\` : ''}` — with no `rel=` attribute (literal or interpolated) emitted in the same tag. Sub-shape (c) exists because callers routinely pass `target: '_blank'` through component-options APIs; without a paired `rel` interpolation the template stays vulnerable whenever the condition is true. Without `noopener`, the new tab can reach back through `window.opener` and redirect the original page (reverse-tabnabbing).<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- `window.open(url, '_blank', featureString)` — when a feature string is passed, modern browsers treat the new window as noopener by default; this is a different API from the `<a target="_blank">` attribute pattern.<br>- Anchors whose `href` is a static same-origin path (starts with `/`, `#`, `./`, or the current origin) AND is visible at the rule-detection site (i.e., not a template-literal `${url}` whose destination is dynamic). Same-origin known-safe destinations don't grant cross-origin capability to the new tab.<br>- `rel="noreferrer"` alone is sufficient (it implies `noopener` in the modern spec). | MEDIUM | Add `rel="noopener noreferrer"` to every `target="_blank"` anchor. HTML strings: `<a href="..." target="_blank" rel="noopener noreferrer">`. DOM construction: after setting `target`, also set `link.rel = 'noopener noreferrer';`. Making this the project-wide default in a link-factory helper is ideal. |
| JSS-04 | `addEventListener('message', handler)` on `window`, `self`, or any EventTarget where the handler body does NOT read `event.origin` (destructured `origin` also counts) and compare it against an expected value before touching `event.data`. Any frame on the page — including an attacker-controlled one in an iframe or a tab that holds a window reference — can send a forged `postMessage` with data that drives the handler's actions.<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- `onmessage` / `addEventListener('message', …)` handlers on a `MessageChannel` **port** (`new MessageChannel().port1` / `.port2`) created in the SAME realm, where neither port is transferred to another realm/document — i.e. NOT a `MessagePort` received via a `window` / `self` `'message'` event's `event.ports`, and neither port is handed to a worker / iframe / `window.postMessage`. A same-realm `MessageChannel` is a closed two-party channel (no external frame can reach it) and `MessagePort` message events carry no security-meaningful `origin` (it's always the empty string), so an origin check is impossible and pointless. Reference safe shape: `_js/nds-loader.js:415-423` (`_yieldChannel` — `port2` is closure-private, used only for `postMessage(null)` as a yield signal; the handler ignores `event.data`).<br>**Still a finding** (do NOT exempt): a `MessagePort` obtained from `event.ports` of a `window` / `self` `'message'` event (that port came from an untrusted sender — validate the originating message's `event.origin` first); any `BroadcastChannel` `onmessage`; a worker / iframe `onmessage` reachable cross-context; or any `MessageChannel` where EITHER port is exposed cross-realm (transferred via `postMessage` or a public NDS API) — once a port leaves the realm the channel is no longer closed, so the handler on the other port is attacker-reachable. | HIGH | Validate origin as the first check in the handler. For a fixed-origin setup: `if (event.origin !== 'https://trusted.example') return;`. For an allowlist: keep a `Set` of allowed origins and early-return on miss. Never compare against `'*'` (no-op), never rely on `event.source` alone (it proves the sender, not the origin), and never skip the check for same-origin-only deployments — same-site subdomains and intermediary frames still cross the origin boundary. |
| JSS-05 | Navigation sinks where the destination is an expression, not a literal: `window.open(x, ...)` / `window.location = x` / `location.href = x` / `location.assign(x)` / `location.replace(x)` / `a.href = x` / `a.setAttribute('href', x)` (the last two when `a` is a newly-created anchor) where `x` is a variable, template literal, or expression whose value at runtime can come from any of these sources: `getAttribute('data-*')`, `dataset.*`, URL query parameters, form-input values, server-response fields, `postMessage` data, caller-supplied component-options object fields (e.g., an `actions: [{ href }]` entry passed to a `create()` API). Without scheme validation, a `data-share-url="javascript:alert(1)"` attribute becomes script execution when the user clicks.<br><br>The rule has two independent concerns. (1) **noopener dimension** — does the new context retain a back-reference to the opener? (2) **scheme dimension** — does the URL parser execute a `javascript:` / `data:` / `vbscript:` payload? Carve-outs apply per dimension; an exemption from one does NOT exempt the other.<br><br>**Explicitly NOT a finding for the noopener dimension** (still subject to the scheme dimension):<br>- `window.open(url, '_blank', featureString)` — when a non-empty feature string is passed, modern browsers default the new context to noopener.<br><br>**Explicitly NOT a finding for either dimension** (do NOT flag):<br>- Navigation to `window.location.href` / `location.pathname` / `location.origin` composed with static paths (e.g., `window.location.origin + '/contact'`). Those values are browser-managed; they can't be injected with `javascript:`.<br>- Values already passed through `encodeURIComponent` for use as query-string parameters (the scheme is whatever the base URL sets, not what the encoded component is).<br>- Values that flow through a same-file scheme-validation helper (`new URL(x, location.href)` checked against an `http:`/`https:`/`mailto:`/`tel:` allowlist) before the navigation sink.<br><br>**Critical**: when `x` traces back to `dataset.*` / `getAttribute('data-*')` / `postMessage.data` / form-input / server-response, the scheme dimension fires regardless of whether the noopener dimension is satisfied by a feature string. Motivating example: `_js/nds-share.js:24-26` calls `window.open(url, '_blank', POPUP_FEATURES)` with `url` from `data-share-url`; noopener is exempt (feature string passed) but scheme is not — `data-share-url="javascript:alert(1)"` flows through and executes. | MEDIUM | Validate the URL's scheme before navigating. Compact form: `try { const u = new URL(x, location.href); if (!['http:', 'https:', 'mailto:', 'tel:'].includes(u.protocol)) return; window.open(u.href, '_blank', 'noopener'); } catch { return; }`. For internal navigation, assert the value begins with `/` or `#` before assigning. For share/copy flows (see `_js/nds-share.js:52–61`), apply the scheme check inside the URL-reading helper so every caller benefits. |
| JSS-06 | **Prototype pollution via JSON-from-writable-store fed into `Object.assign` / object-spread.** A value whose nearest assignment chain reads from a same-origin-writable JSON store — `JSON.parse(localStorage.getItem(...))` / `JSON.parse(sessionStorage.getItem(...))` / `JSON.parse(<cookie-derived>)` / a same-file helper that wraps any of those (e.g. `loadCache(key)` whose body returns `JSON.parse(localStorage.getItem(...))`) — used as a source argument to `Object.assign(target, ..., parsed, ...)` (any source position past the first), or as the source of an object-spread (`{...parsed}`, `{...parsed.settings}`), without a preceding own-key-only filter. Reason: `JSON.parse('{"__proto__":{"polluted":1}}')` produces an own data property named `__proto__`. `Object.assign` (and the spread runtime) calls `[[Set]]` for each source own key, which for `__proto__` triggers `Object.prototype.__proto__`'s setter on the target, replacing the target's `[[Prototype]]` with the attacker-controlled object. Same-origin XSS in any other page on the origin, a malicious browser extension with `storage` access, or a dev-tools paste can plant the poisoned payload; the next page's rehydration takes the prototype assignment.<br><br>Practical impact varies. If the rehydrated object is read only by known-key access (`state.modes`, `state.settings.font-step`) and never spread / iterated via `for...in`, the planted prototype barely changes the component's behavior — the immediate damage is bounded to the rehydrated object. If the object is iterated, spread back into another assign, or used as a key map (`if (key in state)`), the planted properties leak into legitimate logic. Either way, a same-file safe-merge is the cheap defense: it costs one helper call and removes the entire vector.<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- Targets created via `Object.create(null)` — the null prototype means `[[Set]]` for `__proto__` falls through to a regular own data property assignment; the prototype chain is never mutated.<br>- Sources that pass through an own-key filter before the assign — `NDS.i18n.safeMerge(target, parsed)`, an explicit `for (const k of Object.keys(parsed)) { if (k === '__proto__' \|\| k === 'constructor' \|\| k === 'prototype') continue; ... }` loop, or an explicit `delete parsed.__proto__; delete parsed.constructor;` chain immediately preceding the assign.<br>- Annotation exemption (global) naming the writable store AND why the rehydrated payload is trusted (e.g., `// safe: cache writer at L88 only stores values from TYPED_VALUES`).<br>- Multi-source `Object.assign(target, literal1, parsed, literal2, ...)` where every source past the first that comes from a writable store has independently passed the filter exemption above. The rule fires per untrusted source, not per call.<br>- Sources where the writable-store value never reaches `Object.assign` / spread — e.g., parsed payloads consumed only via known-key reads (`if (parsed.foo === 'bar') ...`) and discarded. The vulnerability requires the assign-or-spread step.<br><br>Motivating example: `_js/nds-accessibility.js:153-155` does `const parsed = JSON.parse(raw)` from `localStorage.getItem(STORAGE_KEY)`, then `Object.assign(defaultState(), parsed, { settings: Object.assign(defaultState().settings, parsed.settings \|\| {}) })`. Both Object.assigns take the parsed JSON as a source past the first position with no key filter. Same shape was hardened in the same file at L907 via `NDS.i18n.safeMerge(A11Y_I18N, data.js)` — the load() rehydrator is the remaining gap. | MEDIUM | Route the parsed source through `NDS.i18n.safeMerge(target, parsed)`. The helper at `_js/nds-core.js:164-171` copies own enumerable keys via `Object.keys` and skips `__proto__` / `constructor` / `prototype` — equivalent to `Object.assign` for legitimate keys, hardened against the prototype-key vector. For the multi-source Object.assign shape, replace each writable-store source individually: `Object.assign(target, literal, NDS.i18n.safeMerge({}, parsed), other)` — pass through a fresh own-keys copy. If the consuming object should never have a non-default prototype, switch the target to `Object.create(null)` for full immunity (works whether or not any source is filtered). The helper currently lives under `NDS.i18n` for historical reasons; if non-i18n callers proliferate, surface a JSD-05 promotion candidate to lift it to top-level `NDS.safeMerge`. |

JSS-01 is the highest-false-positive rule in the catalog — static-analysis can't prove that a component-options value is safe at the caller level. When the codebase legitimately trusts a value (e.g., icon class names resolved from a `TYPE_DEFAULTS` enum in the same file), prefer the annotation-exemption comment over widening the rule's numeric/helper allowlist. If the same "safe by construction" rationale repeats across three or more sites, that is a JSD-05 promotion candidate (the rationale becomes a typed enum or helper).
