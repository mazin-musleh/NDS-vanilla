# NDS JS Audit — Performance rules (JSP)

Performance detection catalog for `nds-js-audit`. Read during Phase 2/3 of a `performance` run (or any run with no rule-group filter). Severity (HIGH/MEDIUM/LOW) drives Phase 5 fix order. Audit flow: `SKILL.md`. Other groups: `RULES-JSD.md` / `RULES-JSS.md` / `RULES-JSA.md`. JSD-15 persona canonicals: `PERSONA.md`.

#### JS Performance (JSP)

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| JSP-01 | `addEventListener(['"']resize['"']` on `window` | HIGH | Replace with `NDS.onResize(handler)` — it already debounces at 150ms and pools subscribers.<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- `window.addEventListener('resize', handler)` calls that appear inside a named `bind*()`/`open()` method which is symmetrically paired with a `unbind*()`/`close()` method that calls `window.removeEventListener('resize', handler)`, where the handler is already rate-limited via `NDS.rafThrottle` or `NDS.debounce`. This is a per-open/close lifecycle pattern (the listener's lifetime is bounded by a UI state, not the component's lifetime); NDS.onResize's 150ms debounce would cause visible positioning lag on a live-open UI element. The current add/remove symmetry is the correct approach for this shape. Motivating example: `_js/nds-date-picker.js:945-953` `bindViewportTracking()` / `unbindViewportTracking()` pair. |
| JSP-02 | `addEventListener(['"']scroll['"']` where the handler body does NOT contain any of `requestAnimationFrame(`, `NDS.debounce(`, `NDS.rafThrottle(`, `NDS.onResize(` within ~10 lines of the handler's opening brace. Custom `throttle()` wrappers do not count as throttled unless they themselves use one of these primitives — trace into the wrapper to confirm.<br><br>**Explicitly NOT throttled** (these are findings, not passes):<br>- bare `setTimeout` inside the handler, even with a `clearTimeout`+rebind debounce shape — for scroll, that's JSP-07's job, not "throttled" for JSP-02.<br>- a custom `throttle()` / `rafThrottle()` wrapper whose implementation uses bare `setTimeout` instead of `requestAnimationFrame`.<br>- a handler-body early-return guard on scroll position (e.g., `if (window.scrollY < 200) return;`) — that's a threshold, not a rate limit; the handler still fires on every scroll event above the threshold.<br>- `requestIdleCallback` — it does not help scroll responsiveness and is not on the approved list.<br>- the `passive: true` flag on the listener — that improves scrolling smoothness but does not throttle the handler.<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- Scroll handlers that remove their own listener as part of their work — detectable by a `removeEventListener` reference in the handler body, or in any function the handler calls within ~20 lines of traced call tree. These are single-shot dismiss handlers (e.g., close-on-external-scroll), not hot loops. RAF-throttling adds dismissal latency with no throughput benefit.<br>- Scroll listeners returning a stored unsubscribe handle to the caller, whose body is bounded by a single `Element.contains(...)` / `Element.matches(...)` / `Node.contains(...)` guard before invoking a caller-supplied callback. The dismissal callback typically invokes the returned unsubscribe, so the listener is effectively single-shot per open cycle. RAF throttling would add visible dismissal latency for a scroll-to-dismiss UX. Motivating example: `_js/nds-core.js:669-676` (`NDS.onOutsideScroll`) — handler is `if (scopeEl.contains(e.target)) return; onClose();`, and the returned function at L675 invokes `document.removeEventListener` so callers release the listener as part of their close flow.<br><br>If none of the approved primitives are present in the handler (tracing wrappers where needed), emit the finding. Do not rationalize. | HIGH | Wrap in RAF throttle using `NDS.rafThrottle(fn)`. Copy the pattern from `_js/nds-scroll-more.js:143`. |
| JSP-03 | `new IntersectionObserver`. **Not exempt**: singleton usage, single-element observation, proper `disconnect()`/`unobserve()` disposal, or manual pooling do NOT make this compliant. The rule is about routing through the shared `NDS.onIntersect` pool so observers are grouped by threshold/rootMargin across the whole app — not about leak prevention. A well-disposed bare `new IntersectionObserver` is still a finding. | MEDIUM | Replace with `NDS.onIntersect(el, handler, { threshold, rootMargin })`. Core pools observers by threshold/rootMargin automatically. Store the returned unsubscribe handle. |
| JSP-04 | Three sub-shapes:<br><br>**(a) Raw `new ResizeObserver`.** **Not exempt**: singleton usage, single-element observation, or local `disconnect()` do NOT make this compliant. The rule routes through the shared `NDS.onElementResize` pool; a local pool replicating the same logic is a duplication, not a pass.<br><br>**(b) Pool-routed subscription with discarded unsubscribe handle.** A call to any pool helper that returns an unsubscribe function — currently the full family `NDS.onResize`, `NDS.onElementResize`, `NDS.onIntersect`, `NDS.onOutsideScroll`, `NDS.onAttrChange`, `NDS.onDOMAdd`, `NDS.onDOMRemove`, `NDS.onChildrenChange` — used as a statement-expression where the returned unsubscribe function is not stored on any instance / closure / module-scope reference. When the call sits inside a per-instance or re-callable function (constructor, public `init()`, `setup*()` called per re-init) and the owning component lacks a teardown path that could have invoked the handle, the per-element callback in the pool's map leaks for the process lifetime; on re-init, fresh callbacks stack on top. The pool's selector+fn dedup (e.g., `nds-core.js:205`'s `some(s => s.sel === sel && s.fn === fn)`) does NOT catch fresh closures across instances, because each instance creates a new function reference. Detection: any pool-helper call NOT on the right-hand side of an assignment AND not the trailing expression of a function whose return value is captured by the caller. Mirrors how JSP-08 complements JSP-05 for MutationObserver teardown.<br><br>**Explicitly NOT a finding** (sub-shape (b) only):<br>- Calls inside a singleton-bootstrap IIFE that runs once and is intended to live for the page lifetime (e.g., process-level utility wiring to `document.body`).<br>- Calls inside a public `init()` / `setup*()` guarded by a one-shot latch (`if (_initDone) return; _initDone = true;` flag, `window.ndsXxxInitialized` sentinel, equivalent). The guard makes re-call a no-op so loader re-invocation cannot stack subscriptions. See `_js/nds-theme.js:67-70` (`_initDone`) and `_js/nds-tables.js:414-449` (`window.ndsXxxInitialized`).<br>- Calls inside a public `init()` / `setup*()` targeting ONLY the two dedup-capable pool helpers AND passing a stable, instance-independent function reference: `NDS.onElementResize` (Set-based per-element dedup) and `NDS.onIntersect` (`Map.set` replaces per element). Inline arrows do NOT qualify (each call creates a new reference). **The other six pool helpers** (`NDS.onResize`, `NDS.onDOMAdd`, `NDS.onDOMRemove`, `NDS.onChildrenChange`, `NDS.onAttrChange`, `NDS.onOutsideScroll`) **unconditionally push to their subs arrays with no identity check** — a stable reference does NOT save them; use the init-guard exemption above or the stored-handle pattern at `_js/nds-stepper.js:331-332,364-365`.<br>- Calls where the host element is statically known to outlive the page (`document.documentElement`, `document.body`).<br>- Calls inside an `NDS.onDOMAdd` callback paired with a matching `NDS.onDOMRemove` clean-up — element-lifecycle teardown is handled by the remove handler.<br>- Calls preceded within 3 lines by an annotation comment naming why the per-instance closure is acceptable (annotation-exemption shape mirrors JSP-05 / JSS-01).<br><br>**(c) Destroy bypasses the lifecycle method that releases stored `NDS.on*` handles.** When a component stores pooled-subscription handles inside an `open()` / `show()` / `enter()` lifecycle method (e.g., `this._offScroll = NDS.onOutsideScroll(scope, …)`, `this._unsubResize = NDS.onResize(…)`) AND releases them inside the matching `close()` / `hide()` / `exit()` method, but the component's `destroy()` / teardown path does NOT first call that lifecycle method when the component is in the open state, destroying an already-open component leaves the pooled subscription wired forever. The portal/state side effects (e.g., a menu still attached to `<body>` via `NDS.portal`) compound the leak: the closure is rooted by the pool, the portaled DOM is rooted by the closure. Sub-shape (b) doesn't fire (the handles ARE stored), and sub-shape (a) doesn't apply (no raw observer). Detection: identify a `destroy()` / `teardown()` / `dispose()` method that does NOT reference the close-lifecycle method by name, AND verify the open-state handle assignments live inside `open()` / `show()` rather than the constructor. Severity stays MEDIUM because the leak only fires on the rarer "destroy while open" path — but each occurrence permanently strands a pooled subscription.<br><br>**Raw-listener variant.** The same hazard applies when a component pairs `bindXxx()` / `unbindXxx()` methods around raw `window.addEventListener` / `document.addEventListener` calls (typically rate-limited via `NDS.rafThrottle` to satisfy the JSP-01 / JSP-02 live-positioning carve-outs), invoked from `open()` / `close()` respectively. If `destroy()` does not reference `unbindXxx()` (directly or via the close path), the raw listener leaks for the page lifetime when destroy fires while open. Detection: a `bindXxx()` / `unbindXxx()` pair attaching window/document listeners, AND `destroy()` does NOT call `unbindXxx()` or the close-lifecycle method.<br><br>**Explicitly NOT a finding** (sub-shape (c) only):<br>- Constructor-attached subscriptions released in `destroy()` directly (already covered by sub-shape (b)'s positive case).<br>- `destroy()` calls the close-lifecycle method (`if (this.isOpen) this.close();`) before teardown work — release path runs through the existing close handler.<br>- `destroy()` aborts an `AbortController` that scopes the open-state subscriptions (rare with pooled subs since `NDS.on*` doesn't accept `signal`). | MEDIUM | Sub-shape (a): replace with `NDS.onElementResize(el, handler)`. Returns an unsubscribe function; store it for teardown. Sub-shape (b): capture the returned handle on the instance / closure / module scope (e.g., `this._offResize = NDS.onElementResize(el, handler);` or `let _off; ... if (_off) _off(); _off = NDS.onAttrChange(...);`) and invoke it in `destroy()` / `cleanup()` / re-init. For module-level singletons whose `init()` may run more than once, either guard the registration behind a `let _wired = false;` flag, or capture the handle and release the prior one before re-registering. Motivating example: `_js/nds-stepper.js:329, :361` and `_js/nds-progress.js:59` register `NDS.onAttrChange` inside their public `init()` and discard the unsubscribe; if the loader re-invokes `init()`, the closures stack. Sub-shape (c): make `destroy()` drive the close-lifecycle method when the component is open — `if (this.isOpen) this.close();` as the first statement of `destroy()`. The close path already releases the handles and reverses any portal/state side effects, so destroy stays a thin wrapper. Alternatively, lift the open-state subscriptions to instance-lifetime (constructor) and gate them with an `if (!this.isOpen) return;` inside the handler — but that pays the always-on subscription cost for the rare-open case, so the lifecycle-driven path is preferred. Motivating example: `_js/nds-dropmenu.js:620-626` `destroy()` does `replaceWith(clone)` but does NOT call `close()` first; `this._offScroll` (an `NDS.onOutsideScroll` document-scroll subscription) and `this._unsubResize` (an `NDS.onResize` subscription), both set inside `open()` at L489-490, leak whenever destroy hits while the menu is open. Raw-listener variant: same fix — `destroy()` prepends `if (this.isOpen) this.close();` or calls the matching `unbindXxx()` directly before any teardown work. |
| JSP-05 | `new MutationObserver`.<br><br>**Not exempt**: observing a single element, attaching once at setup, or `observer.disconnect()` on teardown do NOT make this compliant. One MutationObserver per component scales linearly with component count; the NDS pool keeps one observer per configuration and dispatches by selector.<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- MutationObserver instances preceded (within 3 lines above the `new MutationObserver` line) by an inline comment that explicitly rationalizes why the shared `NDS.onDOMAdd` / `NDS.onDOMRemove` / `NDS.onAttrChange` pool does not cover the use case — e.g., content-mutation semantics (`characterData`, text-node observation), cross-frame observation, or other mutation types not in the pool's API.<br>- A bare "// MutationObserver" or unrelated contextual comment does NOT count — the comment must reference the pool's limitation and explain why the local observer is necessary. This is an escape hatch for cases the pool genuinely doesn't cover; it is not an opt-out for "I didn't bother". | MEDIUM | Replace with `NDS.onDOMAdd(selector, handler)`, `NDS.onDOMRemove(selector, handler)`, or `NDS.onAttrChange(selector, attrs, handler)` depending on mutation type. |
| JSP-06 | `addEventListener` without a matching `removeEventListener` or a stored unsubscribe returned by `NDS.on*`.<br><br>**(a) Loop pattern**: listeners attached per-element inside `forEach`/`map`/`for`, no `AbortController` / stored-handle bookkeeping. Flag the whole loop as a single finding with the listener count (not one per line).<br><br>**(b) Dynamic re-attach pattern**: a function attaches per-element listeners, is called more than once per lifecycle (event handlers, re-init paths, data/dir/lang re-renders), AND does NOT detach prior listeners before re-attaching. Two sub-shapes:<br>  **(b.1) forEach/for-loop shape** — attachments live inside a `forEach`/`map`/`for` loop.<br>  **(b.2) Sequential addEventListener shape** — sequential `addEventListener` calls on freshly-rebuilt nodes (typically inside `render()`/`redraw()`/`rebuild()` that does `el.innerHTML = ''` then re-creates children). Listeners attached via `this._ac.signal` where the controller is aborted ONLY at `destroy()` are STILL a finding: the `AbortSignal` retains references across re-renders until destroy fires, so the abort-handler chain grows linearly. Detection: a method that (i) is called more than once per lifetime, (ii) attaches per-element listeners (with or without `this._ac.signal`), AND (iii) does NOT reset the controller (`this._ac.abort(); this._ac = new AbortController()`) before re-attaching.<br>In both sub-shapes, listeners stack per invocation.<br><br>**Explicitly NOT a finding** (sub-shape (b.2) only):<br>- Methods that reset their controller before re-attaching (e.g., `if (this._renderAC) this._renderAC.abort(); this._renderAC = new AbortController();` at the top).<br>- Per-render controllers stored on the rebuilt element (`overlay._ac = new AbortController()`) where the prior overlay's controller is aborted before `innerHTML = ''`.<br>- Single-render components — methods called exactly once per lifetime. If not re-callable, no stacking.<br><br>**(c) Per-instance document/window listener**: a single `document.addEventListener(...)` / `window.addEventListener(...)` inside a constructor or instance-`init()` / `setup*()`, where the handler is NOT wrapped with `{ signal }` AND no stored reference enables later `removeEventListener`. Scales O(n) with component count and leaks per reinit.<br><br>**Explicitly NOT a finding**:<br>- forEach/for loops whose parent container is cleared via `innerHTML = ''` / `replaceChildren()` earlier in the same chain — removed DOM takes its listeners with it.<br>- Sub-shape (c) inside a module-level IIFE / singleton-bootstrap guarded by `window.ndsXxxInitialized` or equivalent one-shot latch (see `_js/nds-tables.js:578, 591`). One-per-page listeners, not one-per-instance.<br>- Sub-shape (c) at the **top level of the module IIFE itself** — straight-line code that runs as the IIFE body. The IIFE is its own one-shot latch (cannot execute twice by construction). See `_js/nds-fontLoading.js:209`.<br>- Sub-shape (c) using `{ signal }` from a **module-scope singleton AbortController** created once at module load and never re-created — purely structural sugar to satisfy the "must use signal" form, even with no `.abort()` call. Detection signature: a module-scope `let`/`const` holding an `AbortController` (`new AbortController()`), created once (lazily or at load) and used in exactly one place to `{ signal }` a single document/window listener, with no `.abort()` anywhere. See `_js/nds-stepper.js:9` (`_stepperControlsAbortController` — declared module-scope, created once behind a one-shot latch in `setupControls()` at L256-257, signal-wired at L286). NOT covered: per-instance `this._ac` — those MUST be aborted in `destroy()`.<br><br>**Sub-shape (a) factory variant** — missing `destroy()` path: factory components with forEach/for listener loops in constructor/init, NO `this._ac`, NO public `destroy()` method. The init guard (e.g., `data-nds-foo-initialized`) prevents stacking per page load, but consumers who remove and recreate the component cannot clean up. Both conditions required: (1) listeners in a loop with no AbortController signal, (2) no `destroy()` anywhere in the file. Motivating example: `_js/nds-tooltip.js:141-152`.<br><br>**Explicitly NOT a finding** (factory-variant only): listeners bound DIRECTLY to content elements (`el.addEventListener(...)` where `el` is the swept element itself, not a stable parent or an outliving closure target) AND each element carries a per-element one-shot init guard (`el._ndsXxxInitialized` JS property or `data-nds-<name>-initialized` attribute set on first visit). Such listeners are reclaimed by GC when the element leaves the DOM, and the per-element guard prevents re-attachment on re-sweep — so there is no stacking and no leak, even without a `destroy()`. The factory-variant targets the case where teardown is genuinely impossible (listeners rooted by a surviving closure, or re-attached without a guard); element-bound + guarded listeners self-clean. Motivating exemplar: `_js/nds-forms.js` `initializeInput` attaches ~10 listeners per `.nds-form-control` input, each guarded by `input._ndsInitialized` — element-scoped, GC-reclaimed on removal, not a finding. | MEDIUM | Sub-shapes (a) and (b): store the unsubscribe handle (`wrapper._offResize = NDS.onResize(...)`), or attach an `AbortController.signal` to every listener so `.abort()` cleans atomically. For dynamic re-attach, abort the previous controller before creating a new one, or use delegated listeners on a stable parent. Sub-shape (c): add `this._ac = new AbortController()` to the constructor, pass `{ signal: this._ac.signal }` to the listener, abort in `destroy()`. See `_js/nds-tabs.js:42, 127, 320`. |
| JSP-07 | **Trailing-edge debounce only.** A wrapper of shape `(...args) => { clearTimeout(t); t = setTimeout(() => fn(...args), CONST_MS); }` (or its named-function equivalent) that is invoked from an event handler / DOM listener / repeated user-input path AND uses a constant ms argument. The intent is "fire `fn` once, after the last call settles" — exactly the shape `NDS.debounce(fn, ms)` wraps. Detection requires: (1) a `clearTimeout(<varName>)` immediately followed by an assignment of `setTimeout(...)` to the same `<varName>`, (2) a constant numeric ms literal as the setTimeout's second arg, AND (3) the enclosing function is callable per-event (not orchestration code).<br><br>**Explicitly NOT a finding** (do NOT flag — these match the surface pattern but the recommended fix doesn't apply):<br>- **Single-pending-action latches.** `clearTimeout` + `setTimeout` calls used as a manual one-shot scheduler — caller invokes `schedule()` and `cancel()` explicitly to orchestrate animations / timing-dependent state, not to throttle event-handler invocations. NDS.debounce returns a wrapper that callers invoke per event; replacing the latch's internal `setTimeout` with `NDS.debounce` would change call-site semantics. Recognized by: one explicit `cancelXxx()` / clear function paired with a `scheduleXxx()` / set function, with at least one caller invoking `cancelXxx()` separately from `scheduleXxx()`. Motivating example: `_js/nds-mainnav.js:88-92` (`scheduleToggleAction` / `cancelToggleAction` pair).<br>- **Priority-routed schedulers.** `setTimeout` whose ms argument is dispatched between values based on a priority/option arg, OR whose dispatch routes to `requestAnimationFrame` for one branch and `setTimeout(fn, ms)` for others. NDS.debounce models a single constant ms; wrapping the dispatcher in `NDS.debounce(fn, ms)` collapses the priority branching. Motivating example: `_js/nds-mainnav.js:304-316` (`overflow.schedule` with `'immediate'` → RAF, `'high'` → 10ms, `'low'` → 100ms).<br>- **Single-shot delay timers (no `clearTimeout` in the same flow).** `setTimeout(fn, ms)` used as a one-time delayed effect (e.g., voice-input timeout, hold-to-repeat acceleration, placeholder restore) where no preceding `clearTimeout` resets the timer. The detector's first-condition gate already excludes these; called out here so they're not flagged on a careless re-read.<br>- **Variable-ms intervals** — `setTimeout(check, nextCheckDelay)` / similar adaptive-polling shapes. The ms is not constant; not a debounce.<br><br>If detection ever uplifts to MEDIUM-style judgment, prefer a SKIP entry (with the specific orchestration shape named) over flagging the wrong pattern — JSP-07's value comes from being precise about what `NDS.debounce` actually replaces. | LOW | Use `NDS.debounce(fn, ms)` — store the wrapper once on the instance/closure and call it per event so the `clearTimeout` bookkeeping stays inside the helper. For single-pending-action latches and priority-routed schedulers, leave the inline pattern in place; consider extracting a richer scheduler primitive into core only if the same shape recurs in 3+ files. |
| JSP-08 | **MutationObserver without disconnect on lifecycle boundary.** A local `new MutationObserver(...)` bound to a specific element (per-element lifecycle, not `document.body` / `document.documentElement`) where EITHER (a) the owning component has a `cleanup()` / `destroy()` / `reinit()` / `teardown()` path that does NOT invoke `.disconnect()` on the observer, OR (b) the component has no teardown path at all — so the observer's lifetime is effectively unbounded despite its element binding. Complements JSP-05: JSP-05 covers pool-routing (is a local observer justified at all?); JSP-08 covers teardown hygiene (once local, is it cleaned up?).<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- Observers stored in an instance array (e.g., `this.observers.push(observer)`) AND disconnected in the teardown path via a loop over that array.<br>- Observers on `document.body` or `document.documentElement` that are intentionally process-lifetime (singleton bootstrap in core or a root-level utility) — not tied to an element that can be re-rendered.<br>- Observers whose JSP-05 exemption comment additionally states that the observer must outlive the element (rare; require explicit rationale, not just absence of disconnect). | MEDIUM | Store the observer handle on the instance (e.g., `this.observers.push(observer)`) and invoke `.disconnect()` in the teardown path. If the owning component lacks a teardown path but creates observers per element, either add the teardown path or re-evaluate whether the observer should be pool-routed via `NDS.onAttrChange` / `NDS.onDOMAdd` / `NDS.onDOMRemove`. |
| JSP-09 | **`onDOMAdd`-wired listener without `onDOMRemove` teardown.** A per-element listener (click / keydown / change / input / pointer*) attached inside an `NDS.onDOMAdd('selector', callback)` handler, where no matching `NDS.onDOMRemove('selector', ...)` exists in the same module to unwire or neutralize the listener when a matched element is removed. Per-element re-wiring guards (e.g., `if (el.hasAttribute(WIRED_ATTR)) return; el.setAttribute(WIRED_ATTR, '')`) prevent double-wiring on re-add but do NOT detach the original listener when the element leaves the DOM — the GC reclaims the element, but if the same element is re-added to a different parent, the guard short-circuits re-wiring and the old closure state persists; if a fresh element takes the same selector, the old closure is still rooted via any captured references. Complements JSP-06: JSP-06 covers per-instance listener stacking; JSP-09 covers observer-wired listener stacking.<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- `onDOMAdd` callbacks that attach via an `AbortController` tied to the element's lifecycle (e.g., `el._ac = new AbortController(); el.addEventListener(..., { signal: el._ac.signal })`) with a paired `onDOMRemove` that calls `el._ac.abort()`.<br>- `onDOMAdd` callbacks that only read or mutate attributes/styles on the matched element (no `addEventListener`) — nothing to tear down.<br>- Listeners attached to a stable parent (event delegation) from inside the callback, where the parent's lifetime is process-long.<br>- `onDOMAdd` callbacks whose body invokes a module-level refresh / re-init routine (not per-element wiring inline), where the invoked routine is idempotent via its own guards — a `_ndsFooInitialized` / `data-nds-foo-initialized` marker on touched elements, an instance-level observer flag (`element._ndsFilterObserver = true`), or a whole-scope setup function that re-queries and rebinds as needed (e.g., `initFormControlClasses()`, `initInputAutoFill()`). These patterns delegate to a refresh-rather-than-per-element shape; the callback body does not itself carry the `addEventListener` the rule targets. See `_js/nds-filter.js:1305` and `_js/nds-forms.js:1657–1658` for the pattern. NOTE: this exemption applies only when the invoked routine genuinely self-guards. If the routine re-attaches listeners unconditionally to the same elements on every invocation, the exemption does not apply and the finding stands. | LOW | Pair the `NDS.onDOMAdd('selector', ...)` subscription with an `NDS.onDOMRemove('selector', ...)` that removes or aborts the attached listener. If the element owns its own AbortController, abort it in the remove handler. If listeners are intrinsic to the element's content, prefer event delegation on a stable parent so no per-element wiring is needed. Motivating example: `_js/nds-cooldown-button.js:191–194` wires `click` on each `.nds-cooldown-button` in an `onDOMAdd` callback with no `onDOMRemove` counterpart. |
| JSP-10 | **Eager module-IIFE DOM work in loader-registered components.** A `_js/nds-*.js` file whose namespace (`NDS.<Name>`) appears in `_js/nds-loader.js`'s `COMPONENTS` array, AND whose outermost module IIFE body — straight-line code directly inside `(() => { ... })()` at module top — contains a DOM-walking query (`document.querySelectorAll(...)`, `getElementsByClassName(...)`, `getElementsByTagName(...)`, `NDS.queryAll(document, ...)`), AND iterates via `.forEach(...)`, `for...of`, or `for(let i = 0; i < ...; i++)`, where the body does at least one of: `.style.*` assignment, `addEventListener(...)`, `setAttribute(...)`, `getBoundingClientRect()`, `new IntersectionObserver(...)`, `new MutationObserver(...)`, `NDS.on*(...)` subscription, `appendChild/removeChild(...)`, `innerHTML/outerHTML = ...`, `.classList.add/remove/toggle(...)`, or CSS-triggering attribute toggle (`hidden` / `data-state` / `data-status`).<br><br>"Top-level of the IIFE body" means NOT nested inside a named function, an arrow assigned to a method, a method on an object literal, a class body method, or any callback passed to `addEventListener` / `requestAnimationFrame` / `requestIdleCallback` / `NDS.onIdle` / `setTimeout`.<br><br>Why it hurts TBT: bundle parse+execute is a single browser task. IIFE-top code runs INSIDE that task — before `_js/nds-loader.js`'s budget-slicer starts. A DOM walk + per-element loop at IIFE top pushes the task past TBT's 50ms long-task threshold. The loader's `initBudgetMs: 5` yields between components during the critical pass but cannot manage code that runs before it.<br><br>**Explicitly NOT a finding**:<br>- Queries inside `NDS.onIdle(...)` / `requestIdleCallback(...)` / `requestAnimationFrame(...)` callbacks at the top of the IIFE — walk runs off the bundle eval task.<br>- Single `document.querySelector(singleSelector)` (NOTE: not querySelectorAll) as a presence guard for early-exit (`if (!document.querySelector('.nds-foo')) return;`). One query is not a walk.<br>- Top-level constant/regex/lookup-table definitions (no DOM walking). Bundle-eval cost but out of this rule's scope.<br>- Files NOT in `_js/nds-loader.js`'s `COMPONENTS` array. Utility modules are exempt; self-attach-without-register problems route to JSP-06(c) / JSD-07.<br>- **Annotation exemption**: comment within 3 lines naming why the work must run at module load (e.g., `// FOUT-blocking — must run before next paint`).
