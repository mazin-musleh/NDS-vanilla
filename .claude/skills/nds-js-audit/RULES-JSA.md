# NDS JS Audit — Architecture rules (JSA)

Per-file architecture/design catalog for `nds-js-audit`. Read during Phase 2/3 of an `architecture` run (or any run with no rule-group filter). JSA findings carry a `Type:` field (`actionable` / `tradeoff`) and a per-rule `Mode` (`greppable` / `deep-read`) — both explained below. Which rules run in full-tree vs single-file is governed by `SKILL.md` Phase 1 (‘JSA deep-read rules in full-tree mode’); full-tree runs skip deep-read rules with the Phase 4 banner. Audit flow: `SKILL.md`.

#### JS Architecture (JSA)

JSA covers per-file design decisions that the per-pattern catalog (JSP/JSD/JSS) cannot generalize — scheduling-primitive choice, unbounded data-structure growth, DOM-API tradeoffs, in-file DRY/KISS, code organization. JSA findings carry plain HIGH/MEDIUM/LOW severity (symmetrical with JSP/JSD/JSS) PLUS a `Type:` field that's unique to this catalog — either `actionable` (concrete change, bounded risk; include a `Fix:` line) or `tradeoff` (a deliberate design choice that no mechanical fix resolves). `fix all` and `fix JSA` apply actionable findings only; tradeoffs are never auto-fixed. But a tradeoff is NOT a dead-end: when a real, higher-leverage way to overcome it exists, the finding carries a `Proposed solution:` line (the lever) plus a `Benefit vs cost:` one-liner, and — when the benefit clearly outweighs the cost — is marked `Type: tradeoff → recommend solving` so the report's Recommended action surfaces it as a focused-refactor candidate (a solution too large for an inline fix can be saved as a plan under `~/.claude/plans/` and linked). Tradeoffs with no better option (or where cost ≫ benefit) carry only `Rationale for accepting:` — never manufacture a `Proposed solution:` to fill the slot.

**Name the axis on `Benefit vs cost:`.** Lead each side of the one-liner with a bracketed axis tag drawn from a closed vocabulary — `init-cost` / `INP` / `memory` / `maintainability` / `cross-file-reach` / `bundle-bytes` — so tradeoffs are scannable and comparable and the accounting stays honest, e.g. `Benefit vs cost: [memory + maintainability] closes a latent per-instance leak + uniform teardown vs [bundle-bytes] ~4 lines, zero call-site change`. This disciplines the EXISTING free-text line — it does not add a second line, and JSP/JSD/JSS findings keep their plain format (only JSA carries the `Type:` / tradeoff machinery).

Mode coverage is split: **greppable** rules run in both full-tree and single-file modes; **deep-read** rules run in single-file mode only (full-tree skips with the mandatory banner described in "JSA deep-read rules in full-tree mode" above). The Mode column documents per-rule coverage.

JSA findings carry the unique `Type:` field — `actionable` or `tradeoff`. Architecture observations frequently surface deliberate design choices the report should acknowledge without forcing a fix; `Type: tradeoff` is the explicit acknowledgment, and Phase 5 batches skip every tradeoff finding regardless of severity. When in doubt, prefer `Type: tradeoff` and explain what was traded — the user can re-classify if they disagree.

**Don't let `tradeoff` become a graveyard.** Prefer `tradeoff` over inventing a fix, but for every tradeoff ask: is there a real, high-leverage way to overcome it? If yes, name it in a `Proposed solution:` line and weigh it in `Benefit vs cost:` — then let the user decide; do not silently bury a solvable cost under "accepted." If the solution's benefit clearly outweighs its cost, mark the finding `Type: tradeoff → recommend solving` and call it out in the Phase 4 Recommended action. A solution too large to apply inline (e.g. a component-wide refactor) should be captured as a plan under `~/.claude/plans/` and linked from the finding, so it isn't lost. Only when there is genuinely no better option, or the cost dwarfs the benefit, does a tradeoff stay a plain `Rationale for accepting:` with no proposed solution.

**Mode coverage discipline.** Greppable JSA rules (JSA-01, JSA-02, JSA-04, JSA-05, JSA-06, JSA-07, JSA-16) run in both full-tree and single-file modes. Deep-read JSA rules (JSA-03, JSA-08, JSA-09, JSA-10, JSA-11, JSA-12, JSA-13, JSA-14, JSA-15, JSA-17, JSA-18, JSA-19) run in single-file mode only — full-tree runs skip them and MUST print the banner from the "JSA deep-read rules in full-tree mode" section above. Silent omission is forbidden; the banner is the report's honest disclosure of what wasn't covered.

JSA-10, JSA-11, JSA-12, JSA-14, JSA-19 have the highest false-positive risk in JSA. Skip the finding if a sub-shape match has a defensible rationale in context (e.g., an outlier-complexity function whose branching IS the documented architectural choice; a "single-strategy abstraction" whose other strategies are deliberately reserved for an upcoming feature; a JSA-14 "leaner" rewrite that can't prove all three invariants, or whose verbosity is load-bearing clarity the reader needs). JSA-14 in particular must NEVER reward code-golf or a new abstraction — when its three invariants (behavior-identical, perf-neutral-or-better, net-simpler-no-new-abstraction) can't all be shown concretely, record a Gap observation, not a finding. When in doubt, record as a Gap observation rather than a finding — JSA earns its keep by sharpening over time, not by padding the report. Two-to-five JSA observations is the typical single-file output; ten+ is a sign of padding.
