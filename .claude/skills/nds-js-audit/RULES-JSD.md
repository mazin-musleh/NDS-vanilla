# NDS JS Audit — DRY/KISS + consistency rules (JSD)

DRY/KISS and cross-file-consistency detection catalog for `nds-js-audit`. Read during Phase 2/3 of a `dry` run (or any run with no rule-group filter). **JSD-05** (cross-file body dedup) needs the full tree — single-file `dry` skips it with the Phase 4 banner. **JSD-15** applies the canonicals in `PERSONA.md` (sibling file) per-file. Severity drives Phase 5 fix order. Audit flow: `SKILL.md`.

#### JS DRY/KISS (JSD)

| ID | What to detect | Severity | Fix |
|---|---|---|---|
| JSD-01 | Three sub-shapes. **(a) Direct `data-state` access** — `setAttribute('data-state', ...)`, `getAttribute('data-state')`, `removeAttribute('data-state')`, `hasAttribute('data-state')`, or assignment to `dataset.state`. The `hasAttribute` form bypasses `NDS.State` just like the others — existence reads still encode the same coupling.<br><br>**(b) `data-*-state` parallel state-machine attributes** — `setAttribute('data-<prefix>-state', ...)`, `getAttribute('data-<prefix>-state')`, `removeAttribute('data-<prefix>-state')`, `hasAttribute('data-<prefix>-state')`, or `dataset.<prefix>State` assignment, where the written/read values match a state-machine vocabulary: `open` / `closing` / `opening` / `closed` / `show` / `hide` / `shown` / `hidden` / `active` / `inactive` / `selected` / `expanded` / `collapsed`. A `data-<prefix>-state` attribute (e.g., `data-toast-state`, `data-menu-state`, `data-overlay-state`) is a parallel state machine that should consolidate onto `data-state` so a single `NDS.State` API serves the whole component — keeping multiple state attributes per element fragments the lifecycle and forces every reader to learn which attribute means what. Motivating example: `_js/nds-alert.js:189,261` writes `data-toast-state` with `'show'`/`'hide'`. Carve-out: `data-<prefix>` attributes whose values are NOT in the state-machine vocabulary (e.g., `data-position-vertical="top"`, `data-axis="x"`, `data-variant="primary"`) are configuration / positional / classification attributes, NOT state.<br><br>**(c) Attribute-selector queries against `data-state`** — `querySelector(...)`, `querySelectorAll(...)`, `matches(...)`, `closest(...)` whose selector string contains `[data-state~="<token>"]`, `[data-state="<token>"]`, `[data-state*="<token>"]`, or any other attribute-selector form targeting `data-state`, AND the embedded token matches the state-machine vocabulary listed under sub-shape (b) (`open` / `closing` / `opening` / `closed` / `show` / `hide` / `shown` / `hidden` / `active` / `inactive` / `selected` / `expanded` / `collapsed`). HTML-template writes that embed `data-state="<token>"` as a literal attribute inside an `innerHTML` string (e.g., `el.innerHTML = '<button data-state="active">'`) also count under (c) — the template bypasses `NDS.State.set` just like sub-shape (a)'s direct write. The selector form is the read-side analog of `hasAttribute('data-state')`: it queries by state token in a way that couples consumer code to the State vocabulary outside the State API, just as the `hasAttribute` form encodes existence-by-token outside the API. **Carve-out (vocab gate, sub-shape (c) only):** tokens NOT in the state-machine vocabulary list above — component-private classification tokens such as `today`, `other-month`, `sorted-asc`, `sorted-desc`, `range-start` — are NOT findings even when embedded in `[data-state~="<non-vocab-token>"]` selectors. JSD-01(c) is bounded to state-machine vocab by design; non-vocab tokens are component-private and the file owns both write and read, so the coupling is internal. Exemplar: `_js/nds-date-picker.js:1505-1509` documents the distinction inline (`'selected' / 'range-start' are state-vocab tokens → NDS.State.has; 'today' / 'other-month' are component-private → selectors stay as-is`). Carve-out: selectors targeting custom `data-*` attributes outside the state vocabulary (e.g., `[data-no-auto-close]`, `[data-dropmenu-no-click]`, `[data-position-vertical]`, `[data-filter]`) are configuration markers, not state — NOT a finding. Carve-out: when a component already maintains its own canonical state-tracking data structure (e.g., a module-level `Set` of open instances, like `_js/nds-mainnav.js:_openDropdowns`), the selector form is doubly wrong (it duplicates the source of truth); when no such structure exists yet, the fix may require adding one. All prior exemplars are resolved (refactored to module-level Sets / `NDS.State.has` iteration); the rule stands as a regression guard.<br><br>**Carve-out: server-rendered state-vocab reads (sub-shape (c) only).** When the queried `[data-state~="<token>"]` is set in HTML template output (Liquid / Jekyll-rendered) AND no preceding `NDS.State.set(*, '<token>')` / `NDS.State.add(*, '<token>')` call in the same file writes that token to a matched element type, the selector is the canonical entry-point — it reads initial state from DOM, not duplicate state from a JS Set. NOT a finding. Detection refinement: for each `[data-state~="<token>"]` selector, grep the same file for `NDS.State.set` / `NDS.State.add` writes of that token; if zero writes of that token exist anywhere in the file, the read is server-rendered and exempt. The recommended fix (iterate + `NDS.State.has`) would replace a single browser-optimized selector with a JS-level filter — strictly worse, no behavior change. Motivating examples (both NOT findings under this carve-out): `_js/nds-drawer.js:236` (`initActiveStates` reads server-rendered `li[data-state~="active"]` descendants to mark parent buttons; no `NDS.State.set(*, 'active')` write of `'active'` exists in the file), and `_js/nds-sidemenu.js:157` (`updateToggleLabel` reads the server-rendered active item's label to display on the collapsed toggle; same shape). | HIGH | Sub-shape (a): use `NDS.State.add/remove/has/get/set/clear/apply`. For `hasAttribute('data-state')` reads, use `NDS.State.get(el) !== ''` when testing for any state, or `NDS.State.has(el, 'specific-token')` when a specific token is known. Sub-shape (b): migrate the parallel state machine onto `data-state` using component-namespaced tokens (e.g., `data-toast-state="show"` → `NDS.State.add(toast, 'toast-show')` or `NDS.State.set(toast, 'shown')`). If the component genuinely needs two orthogonal state axes (rare — usually a redesign smell), use space-separated tokens on a single `data-state` rather than two attributes. Sub-shape (c): when iterating a known set, swap the selector for `NDS.State.has(el, token)`; when querying by state across an unknown scope, either track an instance/module `Set` (mirroring `_js/nds-mainnav.js:_openDropdowns`) or omit the attribute from any HTML template and call `NDS.State.set(el, token)` after `innerHTML` is set. If three or more files independently re-implement attribute-selector queries by state, surface as a JSD-05 promotion candidate for a core `NDS.State.queryByToken(scope, sel, token)` helper — the recurring pattern would justify a shared abstraction over per-component Sets. |

| JSD-03 | Hardcoded layout pixels bypassing the token layer, in three sub-shapes: **(a) viewport comparisons** — any integer literal in `[400, 2000]` compared (`<`, `<=`, `>`, `>=`, `===`, `!==`) against `window.innerWidth`, `document.documentElement.clientWidth`, or `document.body.clientWidth`. The range captures the realistic viewport-breakpoint space while excluding unrelated small/large magic numbers (counts, sizes, timeouts). Off-by-one literals like `958` (near-desktop) or `1439` (near-large-desktop) are the main reason for widening beyond the canonical 768/1024/1280/1440 list; **(b) fallback literals** — the canonical breakpoint values (768/1024/1280/1440) used as the right-hand side of `\|\|` after `getComputedStyle().getPropertyValue(...)` parses a CSS variable (e.g., `parseInt(styles.getPropertyValue('--nds-...')) \|\| 1280`); **(c) `matchMedia` breakpoint literals** — any integer literal in `[400, 2000]` embedded inside a string argument to `window.matchMedia(...)` as part of a `min-width` / `max-width` / `min-height` / `max-height` media feature (e.g., `window.matchMedia('(max-width: 600px)').matches`, `window.matchMedia('(min-width: 601px) and (max-width: 960px)')`). Off-by-one literals like `601` / `961` are the common shape (media queries use inclusive edges while `NDS.breakpoints` picks one side as canonical), and they silently drift from the tokens on every re-audit. All three shapes hardcode the token into JS; (b) also hides a missing-CSS-var bug when the fallback silently takes over.<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- Numeric literals on the right-hand side of a module-scope `const` declaration whose identifier is SCREAMING_SNAKE_CASE (e.g., `CONTENT_MAX_WIDTH_FALLBACK`, `SLIDES_MAX_WIDTH`) AND which has a rationale comment within 3 lines above explaining why JS needs the numeric value. This IS the fix for sub-shape (b); re-flagging it on every audit run is noise. The detection fires on raw literals in comparisons or fallback positions, not on named-constant declarations that encode the token-layer shim.<br>- Integer comparisons against viewport width where the literal is OUTSIDE `[400, 2000]` — those are genuinely not breakpoint logic (thresholds for tiny embed widths, very-wide ultra-monitor layouts, etc.) and should be judged case-by-case rather than swept up by this rule.<br>- `matchMedia` strings embedded in feature-query syntax unrelated to viewport width/height (e.g., `(prefers-reduced-motion: reduce)`, `(hover: none)`, `(prefers-color-scheme: dark)`, `(pointer: coarse)`) — sub-shape (c) only catches literal integer pixel values inside `(min\|max)-(width\|height)` media features.<br>- `matchMedia` calls whose argument is `NDS.breakpoints.<token>` or a module-scope `const` with a rationale comment — the token layer is already in use, no finding. | MEDIUM | Viewport comparisons and `matchMedia` literals → use `window.matchMedia(NDS.breakpoints.mobile/tablet/'tablet-max'/desktop/'desktop-max'/'large-desktop').matches`. If the specific pixel threshold isn't in `NDS.breakpoints`, either add it to core or align the component to the closest existing token (e.g., `958` → `'desktop'` at 960px, `601` → `'tablet'` at 600px, `961` → `'desktop'` at 960px). Fallback literals → ensure the CSS variable is always defined in base styles so the fallback never fires, or extract the literal to a SCREAMING_SNAKE_CASE module-scope constant with a rationale comment explaining why JS needs the value. |
| JSD-04 | Re-implemented locale/direction check that duplicates `NDS.lang` / `NDS.isRTL` / `NDS.isArabic`. Two sub-shapes:<br><br>**(a) Direction/RTL checks:** `document.dir`, `document.documentElement.dir === 'rtl'`, `getAttribute('dir')` on `document.documentElement` or `document.body`, or equivalent reads of the `dir` attribute to derive RTL-ness.<br><br>**(b) Language checks:** `document.documentElement.lang`, `document.lang`, `document.documentElement.getAttribute('lang')`, or derivations that lowercase / prefix-split / compare-against-`'ar'` those values to produce a lang code (e.g., `(document.documentElement.lang \|\| 'en').split('-')[0].toLowerCase()`, `document.documentElement.lang.startsWith('ar')`).<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- `NDS.lang` / `NDS.isRTL` / `NDS.isArabic` definitions inside `nds-core.js` — that's the source of truth, not a duplicate.<br>- Per-element `lang` reads: `el.getAttribute('lang')` / `el.lang` where `el` is NOT `document.documentElement` / `document.body`. Component code may legitimately read per-element language overrides (e.g., a widget switching mid-page); that's not a duplication of the document-level `NDS.lang`.<br>- Per-element `direction` reads: `getComputedStyle(el).direction === 'rtl'` (or `=== 'ltr'`) where `el` is NOT `document.documentElement` / `document.body`. `NDS.isRTL` is a document-level signal and cannot answer per-instance direction questions — a component nested inside a `dir="…"` wrapper or styled via CSS `direction:` (e.g., chart canvases, embedded RTL widgets inside an LTR shell) must resolve direction at the element level. Motivating example: `_js/nds-chart.js:404` reads `getComputedStyle(this.el).direction` for per-chart RTL handling — a legitimate use case that `NDS.isRTL` cannot satisfy. | MEDIUM | Use `NDS.isRTL` (live getter) or `NDS.lang` / `NDS.isArabic`. |
| JSD-05 | **Promotion candidate.** A helper function with near-identical signature and body appears in two or more component files and has no equivalent in `nds-core.js`. | MEDIUM | Propose an `NDS.*` API name and list every call site. Typical candidates: focus-trap, scroll-lock, `getComputedStyle` readers, `matches()` / `closest()` helpers, RAF batchers, event-delegation factories, z-index stackers, `aria-expanded` toggles. Surface the candidate in the Phase 4 report; editing `nds-core.js` to add the helper + migrating every call site is a Phase 5 action the user approves per-candidate via `promote <api-name>` (see "Applying promotion candidates" in `FIX.md`). |
| JSD-06 | DOM read after write in the same frame (e.g., `setProperty` then `getBoundingClientRect` in a loop).<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- Standalone forced-reflow triggers (`void el.offsetHeight;` or `el.offsetHeight;` as the sole statement on its line) are deliberate transition-start patterns, not thrashing.<br>- Staged-measurement patterns: a write to `style.cssText` / `style.visibility` / `style.position` that puts an element into a measurable off-screen state, followed by `offsetHeight` / `offsetWidth` / `getBoundingClientRect()` reads. The read semantically depends on the write (the element must be staged before it can be measured), so "batch reads before writes" does not apply. Require an inline comment near the write confirming the staging intent to earn this exemption — otherwise it's indistinguishable from a thrash loop.<br><br>Thrashing is a LOOP pattern (N writes × N reads within one synchronous block). One-shot write-then-read sequences are not thrashing. | LOW | Batch reads before writes: collect every layout read into locals first, then apply all writes. Exemplar: `_js/nds-core.js` `NDS.flipPosition` (~L1046) — collects `triggerRect` / viewport dims into locals before any positional write. |
| JSD-07 | **Component contract conformance.** A `_js/nds-{name}.js` file whose public surface does not match one of the two shapes the loader expects. Factory components must expose `NDS.{Name}.init()`, `NDS.{Name}.reinit()`, and `NDS.{Name}.create(el)`. Singletons must expose `NDS.{Name}.init()` plus their component-specific methods. The namespace `{Name}` must be the PascalCase form of the filename suffix.<br><br>**Explicitly NOT a finding** (utility-module exemption): files that are NOT registered in `_js/nds-loader.js` AND have no DOM-attachment surface (no top-level `document.querySelectorAll` / `document.addEventListener` to a content selector / `NDS.onDOMAdd` subscription) AND publish only pure-function helpers consumed by other components. To verify the exemption applies, grep `_js/nds-loader.js` for the module's namespace literal — if absent from the loader's registry AND the module's IIFE body contains no DOM-attachment calls, the contract does not apply. Example: `_js/nds-voice-recognition.js` exports `{audioFeedback, isSupported, getLanguage, create, startListening, stopListening, _loaded}` for `nds-forms.js` to consume; it is not loader-registered and does not self-attach.<br><br>**Also NOT a finding** (declarative-delegation library exemption): a module that (1) is NOT registered in `_js/nds-loader.js`, (2) publishes a documented programmatic API (`NDS.<Name>.<methods>`), AND (3) whose ONLY DOM-attachment surface is a single lazily-installed document-level *delegated* listener (a `document.addEventListener('click'\|'input'\|…, handler)` whose body resolves targets via `e.target.closest('[data-<name>]')`), optionally paired with an `NDS.onDOMAdd('[data-<name>]', …)` subscription that re-arms the same delegation for late-added triggers. Such modules drive declarative `[data-*]` attributes rather than per-element component instances; they need no `init()`/`reinit()`/`create()`, and adding one nothing calls would itself be a JSD-11 pretender. The defeating signal for the pure-utility exemption above (a single delegated listener + `onDOMAdd` re-arm) is exactly this exemption's *qualifying* signal — the distinction is that the DOM attachment is one document-level delegation, not per-element wiring the loader would need to manage. Motivating example: `_js/nds-export.js` exposes `NDS.Export.{export,csv,xls,pdf,collect,…}` and self-installs one delegated `[data-export]` click listener (L431) plus an `NDS.onDOMAdd('[data-export]', …)` re-arm (L435); not loader-registered. | MEDIUM | Restructure the module to match the shape of the closest peer (factory example: `_js/nds-modal.js`; singleton example: `_js/nds-theme.js`). Inconsistent shapes force every reader to relearn the API per component and block the loader from treating components uniformly. For utility modules covered by the exemption above, no fix is required — but a one-line comment near the namespace export naming the consuming components keeps future audits from re-flagging. For declarative-delegation libraries, a one-line comment near the namespace export naming the pattern (e.g. `// Library + declarative [data-export] delegation; no loader entry by design.`) serves the same purpose. |
| JSD-08 | **Dead defensive guards.** Inside a component registered with `nds-loader.js`, checks like `typeof NDS !== 'undefined'`, `typeof NDS.X !== 'undefined'`, `NDS && NDS.X`, the `window.`-prefixed variants (`window.NDS && window.NDS.X`, `window.NDS && NDS.X`, any mix), OR the optional-chaining variants (`NDS?.X?.method()`, `NDS.X?.method()`, `NDS.X?.subProp`) used as preconditions before calling core APIs. Detection covers bare-`NDS`, `window.NDS`, AND optional-chaining (`?.`) forms — earlier drafts of this rule only caught the bare-`&&` form and missed both fully-prefixed guards in defensive consumer-style code AND the modern `?.` collapse, which is the equivalent existence-guard expressed via optional chaining. The loader guarantees `NDS` and every registered namespace exist before any component's `init()` runs, so these guards are dead code that implies uncertainty the loader contract explicitly denies. Multi-part guards where ONLY the final method-existence check is load-bearing (e.g., `NDS && NDS.Forms && NDS.Forms.validateForm`) collapse to just the final call when the API is unconditionally assigned — verify by grepping `NDS.{Namespace} = {` in the owner module to see if the method lives inside an always-present object literal.<br><br>**Explicitly NOT a finding** (do NOT flag these as JSD-08):<br>- `typeof window !== 'undefined'` — this is a runtime-environment guard protecting against SSR/Node execution, not a loader-contract guard. Whether it's actually dead depends on the project's SSR posture, which is a separate architectural decision outside this rule's scope.<br>- `typeof document !== 'undefined'` — same reasoning.<br>- `NDS.Alert` / `NDS.X` checks in code that runs BEFORE the loader (module top level, or in a bootstrap script) — for those, the loader contract has not yet fired, so the guard is load-bearing.<br>- **Annotation exemption (global — documented soft dependency).** The comment must (1) name the specific dependency by namespace (`NDS.Alert`, `NDS.Forms`, etc.) and (2) state why it's optional (selective bundling, consumer environments that may exclude a component). Motivating example: `_js/nds-cooldown-button.js` ~L105 — `// Soft dependency — component no-ops the toast if Alert isn't bundled.`<br><br>Scope is deliberately narrow on two axes: **(1) where the guard appears** — only loader-contract-redundant guards inside method/function bodies called after the component's `init()` runs; AND **(2) what API the guard protects** — only guards on the `nds-core.js` utility surface, which the loader guarantees is loaded before any component's `init()` runs. The core-utility allowlist (the only namespaces in scope for JSD-08): `NDS.State`, `NDS.Status`, `NDS.breakpoints`, `NDS.lang`, `NDS.isRTL`, `NDS.isArabic`, `NDS.langKey`, `NDS.escapeHtml`, `NDS.debounce`, `NDS.rafThrottle`, `NDS.onResize`, `NDS.onElementResize`, `NDS.onIntersect`, `NDS.onDOMAdd`, `NDS.onDOMRemove`, `NDS.onChildrenChange`, `NDS.onAttrChange`, `NDS.onOutsideScroll`, `NDS.trapFocus`, `NDS.scrollLock`, `NDS.cache`, `NDS.i18n`, `NDS.portal`, `NDS.unportal`, `NDS.placeFixed`, `NDS.flipPosition`, `NDS.uniqueId`, `NDS.transitionSpeed`, `NDS.focusableSel`, `NDS.gridLastRow`, `NDS.closest`, `NDS.queryAll`, `NDS.querySelector`, `NDS.needsPortal`, `NDS.onIdle`.<br><br>**Also in scope of JSD-08 — self-referencing guards on the file's own namespace.** A third in-scope shape: optional-chaining (`NDS.<Same>?.<method>?.()`, `NDS.<Same>?.<prop>`) or `&&` (`NDS.<Same> && NDS.<Same>.<method>`) guards on the *file's own exported namespace* — where `<Same>` matches the file's own `NDS.<Same> = { ... }` definition earlier in the same module. By module-eval time, that assignment has already executed, so the guard is dead in exactly the shape JSD-08 catches for core utilities. Detection: identify the file's primary namespace via a grep for `^\s*NDS\.<Name>\s*=\s*\{` (or `NDS\.<Name>\s*=\s*\(?\s*\{`) at module top level, then flag any subsequent `NDS.<Name>?.<method>?.()` / `NDS.<Name> && NDS.<Name>.<method>` guards in the same file. Same fix (delete the guard) applies. **Carve-outs**: (a) guards inside code paths that run BEFORE the namespace assignment (rare — would require the guard to live above the `NDS.<Same> = { ... }` line); for those, the namespace genuinely doesn't exist yet and the guard is load-bearing. (b) annotation exemption — same comment-rationale shape as the core-utility annotation exemption above. Resolved (was the motivating example): `_js/nds-accessibility.js` migrated `NDS.Accessibility?.init?.()` / `NDS.Accessibility?.toggle?.()` → bare `NDS.Accessibility.init()` (L1097) / `NDS.Accessibility.toggle()` (L1104) inside the `bootAccessibility()` IIFE (`NDS.Accessibility = { ... }` at L1074). The dead self-referencing guards were removed; the rule stands as a regression guard.<br><br>**Explicitly NOT in scope of JSD-08** (route to JSD-12 instead): guards on *sibling-component* namespaces (`NDS.Alert`, `NDS.Backdrop`, `NDS.Copy`, `NDS.Dropmenu`, `NDS.Feedback`, `NDS.Forms`, `NDS.Mainnav`, `NDS.Modal`, `NDS.Pagination`, `NDS.Sort`, `NDS.Tooltip`, `NDS.TimeDate`, `NDS.UserFeedback`, etc. — anything outside the core-utility allowlist above AND not the file's own namespace). The loader does NOT guarantee one sibling exists when another's `init()` runs — deferred components can be absent when the critical pass starts, optional components can be bundle-excluded by consumers, and load order across siblings is registry-order-dependent. A guard on a sibling component is therefore *potentially* load-bearing; it shouldn't be removed automatically, but it should be annotated. JSD-12 catches unannotated sibling guards. | LOW | Remove the guard. If the call site truly cannot assume the dependency (e.g., code that runs before the loader or outside the component lifecycle), route it through the loader or defer the call — do not hedge at the call site. |
| JSD-09 | **Teardown references properties never stored, or stored handles never invoked.** Four sub-shapes:<br><br>**(a) `this.handlers.{name}` map shape** — Inside any teardown path (function named `cleanup` / `destroy` / `teardown`, or any function containing `removeEventListener(..., this.handlers[name])` — either in a loop over named keys OR as straight-line `removeEventListener` calls referencing `this.handlers.{name}` directly), every handler key referenced must have a matching assignment `this.handlers.{name} = ...` somewhere else in the same file. A teardown that dereferences `this.handlers.foo` when `foo` was never stored is dead-code-pretending-to-clean-up: the `removeEventListener` call receives `undefined` and silently does nothing. Detection: for each `this.handlers.{name}` or `this.handlers[name]` reference inside a teardown path, grep the file for `this.handlers.{name} =` or `this.handlers[{name-literal}] =`.<br><br>**(b) Instance-property handler shape** — `removeEventListener(..., <owner>._handlerName)` or `removeEventListener(..., <owner>.<propName>)` (where `<owner>` is `this`, an instance reference, or a closure-scoped variable; `_handlerName` / `propName` is a single identifier, not a method call). Same detection as (a) but generalized: grep the file for `<owner>._handlerName =` / `<owner>.<propName> =`. If no assignment exists, the call receives `undefined` and silently noops. Canonical bad shape: a `removeEventListener('resize', el._someHandler)` line in teardown where the file never executes `el._someHandler = function(...) {...}` before the matching `addEventListener`. Also covers `<owner>.observer.disconnect()` / `<owner>.<prop>.disconnect()` / `<owner>.<prop>()` zero-arg call forms in teardown that depend on `<prop>` being assigned — e.g., `this.observer.disconnect()` where `this.observer` was never assigned in the constructor or setup methods.<br><br>**(c) Stored `NDS.on*` unsubscribe handle never invoked** — `<owner>._off* = NDS.on*(...)` stored on an instance/closure but no teardown path in the same file calls `<owner>._off*()`. The pooled subscriber leaks for the process lifetime; on re-init the same component stacks subscriptions. Detection: collect every `<owner>._off* = NDS.on*(...)` assignment, then grep for `<owner>._off*(` / `<owner>._off*?.()` call forms. If zero call sites exist in any teardown / re-init path, emit the finding.<br><br>**(d) Collection-shaped teardown stores never populated** — An instance field assigned to an empty collection (`<owner>.<prop> = []`, `<owner>.<prop> = new Set()`, `<owner>.<prop> = new Map()`) AND iterated in a teardown path (e.g., `this.observers.forEach(o => o.disconnect())`, `this._cleanups.forEach(fn => fn())`) BUT no `<owner>.<prop>.push(...)` / `<owner>.<prop>.add(...)` / `<owner>.<prop>.set(..., ...)` write to that field exists anywhere in the file. The forEach is a permanent no-op; the field is dead bookkeeping that pretends to release resources nothing ever stored. Sub-shapes (a)–(c) all check for assignment-presence; (d) checks for collection-population, since `this.observers = []` IS an assignment but writes nothing useful. Detection: collect every `<owner>.<prop> = (\[\]\|new (Set\|Map\|Array)\(\))` initializer, then grep the file for `<owner>.<prop>.push(` / `<owner>.<prop>.add(` / `<owner>.<prop>.set(`. If zero population sites exist AND the field is iterated in a teardown path, emit the finding. Synthetic shape: `constructor() { this.observers = []; }` … `cleanup() { this.observers.forEach(o => o.disconnect()); }` with no `this.observers.push(...)` anywhere in the file — the forEach is a permanent no-op. Detection: the 2nd argument is a **bare function identifier** (NOT an inline `nodes => …` / `function (nodes) { … }`) whose definition dereferences its **first parameter as an element** — calls `.getAttribute` / `.setAttribute` / `.querySelector` / `.matches` / `.closest` / `.classList` / `.style` on the param, or hands it to a function expecting a single element.<br><br>**Explicitly NOT a finding** (do NOT flag):<br>- Callbacks that take and iterate the array — `nodes => nodes.forEach(fn)`, `function (nodes) { nodes.forEach(...) }`, or an index loop over the param. The correct shape. | JSD-18 | **Init sentinel stamped on a failed construction.** A factory sweep (`document.querySelectorAll(sel).forEach(...)`) that constructs a per-element controller and then writes the init-guard sentinel — the `data-nds-<name>-initialized` attribute, the `el._ndsXxxInitialized` JS-property form, or the instance expando (`el.ndsFooControls = instance`) — **unconditionally**, while the constructor carries an early-`return` bail path (typically `console.warn('NDS Foo: …')` + `return` on invalid markup). The sentinel then marks work that never happened: the sweep's own `if (!el.hasAttribute('data-nds-<name>-initialized'))` guard skips that element forever, so `reinit()` can never retry it once the missing markup arrives (a `<tbody>` filled by a fetch, a menu whose target renders late). The expando compounds it by handing consumers a half-constructed instance whose fields the bail path never assigned.<br><br>Detection: find each constructor with a guard-then-`return` preceding its `this.init()` call, then read the sweep that instantiates it — a finding when the sentinel write (or the expando assignment) is not conditional on the construction having succeeded.<br><br>**Cross-reference:** PERSONA entry 5 owns the sentinel's *form* (which attribute, which scope); JSD-18 owns *when* it may be written. JSD-05 is the most qualitative rule: require ≥2 distinct files with substantially-overlapping bodies (same shape, same side effects) before flagging, and name the proposed API naturally (`NDS.trapFocus(el)`, not `NDS.helperFn7`).

JSD-14 has the highest false-positive risk in JSD. Skip the finding if a sub-shape match has a defensible rationale in context (e.g., deep nesting that mirrors a state machine the reader needs to see linearly; a redundant guard that documents an invariant the next reader would otherwise miss). When in doubt, record as a Gap observation rather than a finding — the rule earns its keep by sharpening over time, not by padding the report.

JSD-15's false-positive defense is `PERSONA.md` itself. The persona's per-entry carve-outs name the concept-different cases that look like divergence but aren't (transient toasts using `show`/`hide` vs modals using `open`/`close`; the date-picker two-phase `cleanup()`+`destroy()` vs single-method instances; `window.nds<Name>Initialized` flags for tree-wide observers in tables.js). The audit MUST read these carve-outs before flagging — a file matching a documented carve-out's discriminator is NOT a finding. When divergence is real but the canonical itself feels wrong for the case at hand, the answer is a Phase 7 EVOLVE proposal to revise the canonical with the file's reasoning, not a quiet skip — the persona changes through deliberate revision, never through silent erosion.
